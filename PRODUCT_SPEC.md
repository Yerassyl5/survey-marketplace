# PRODUCT_SPEC.md — Продуктовое ТЗ маркетплейса инженерных изысканий

> Этот документ описывает **что строить и в каком порядке**.
> Как устроена система архитектурно — см. `CLAUDE.md`.
> Принцип: каждая волна заканчивается работающим, проверяемым куском продукта. Не начинать следующую волну, пока текущая не закрыта по критериям приёмки.

---

## 0. Контекст и сегменты

Маркетплейс соединяет заказчиков изысканий с проверенными исполнителями в Казахстане. Кастдев проведён, спрос подтверждён.

**Стартовая сторона рынка — исполнители** (доступ к ним есть). Их привлекаем первыми, наполняем ими маркетплейс, затем ведём заказчиков «на витрину, где уже есть подрядчики».

**Сегменты заказчиков и стратегия по каждому:**
- Физлица (построил дом/дачу) — приходят разово. Не удерживаем, хорошо обслуживаем. Это поток новых заявок и сарафан.
- Средние / частые застройщики — **главная цель ретеншна**. Их круг подрядчиков регулярно не справляется.
- Крупные компании с готовым процессом — почти не удерживаются стандартными средствами, не цель на старте.

**Главный продуктовый риск — обход платформы (disintermediation) и низкий ретеншн заказчика.** Подписка-перед-просмотром защищает от обхода со стороны исполнителя. Ретеншн-кабинет (объекты + история + избранное) защищает удержание заказчика.

## ВОЛНА 1 — Ядро маркетплейса (MVP)

Цель: проверить, что обе стороны реально находят друг друга и доходят до сделки. Без ИИ, без тендера, без оплаты внутри.

### 1.1. Регистрация и роли
- Регистрация заказчика и исполнителя, выбор роли.
- При регистрации пользователь выбирает тип лица (юр./физ.) и вводит данные **вручную**:
  - физлицо: ФИО, ИИН, почта, телефон;
  - юрлицо: ФИО (контактное лицо/руководитель), БИН, почта, телефон.
- БИН/ИИН на старте просто сохраняются, без автопроверки и автозаполнения.
- SMS-подтверждение телефона на старте НЕ делаем.
- Профиль исполнителя с полями допусков (`license_number`, `attestation_number`, `license_expiry`, `verification_status`, `verification_method`).
- Профиль-портфолио исполнителя (работает как витрина).
- Вход по email+паролю, выдача токена; `GET /me/` для получения текущего профиля.
**Приёмка:** пользователь выбирает тип лица и регистрируется с ручным вводом данных; исполнитель дополнительно загружает документы; статус по умолчанию «на проверке»; зарегистрированный пользователь может войти и получить токен; защищённые эндпоинты (документы исполнителя, объекты) работают по токену, не только через Django Admin.
**Отложено (см. architecture.md §4.1):** SMS-подтверждение, автопроверка/автозаполнение по БИН (юрлица — реализуемо через `identity_lookup`), автозаполнение по ИИН (физлица — только через Egov/ЭЦП).

### 1.2. Верификация исполнителя (ручная, опциональный процесс)
- Верификация — НЕ часть регистрации. Исполнитель регистрируется и может откликаться сразу,
  будучи неверифицированным; статус верификации просто виден заказчику в каждом отклике.
- Исполнитель по желанию загружает сканы лицензии/аттестата на верификацию.
- Модератор в Django Admin видит документы, ставит «верифицирован» или «отклонён» с причиной.
- Отклонение: причина видна исполнителю в личном кабинете И приходит на почту.
  Документы можно пересдать (статус возвращается в «на проверке»).
**Приёмка:** модератор может верифицировать/отклонить исполнителя с причиной; статус и причина
отклонения видны исполнителю в профиле и на почте; неверифицированный исполнитель откликается
без ограничений; заказчик видит статус верификации в каждом отклике.

### 1.3. Объекты (sites) — заложить сущность
- Заказчик создаёт объект (точка/контур на карте — PostGIS).
- Заявки привязываются к объекту.
**Приёмка:** заявка не существует без привязки к объекту; объект отображается на карте.

### 1.4. Заявка, отклик и выбор исполнителя
- Заказчик создаёт заявку на изыскания (тип работ, описание, файл ТЗ или AI-черновик — см. 1.12,
  привязка к объекту, геометрия). Требует подтверждённой почты (см. 1.11).
- Исполнитель видит ленту заявок в своей нише/городе, откликается (цена, срок, комментарий).
  Требует подтверждённой почты (см. 1.11).
- **Рассмотрение:** заказчик видит отклики (цена, срок, комментарий, статус верификации) и жмёт
  «рассмотреть» на одном или нескольких. Система сразу показывает телефон исполнителя и
  фиксирует факт рассмотрения (кто, кого, когда). Заказчик созванивается вне системы.
- Заказчик выбирает исполнителя **только среди рассмотренных** (минимум один рассмотренный
  обязателен).
- **Отзыв заявки:** заказчик может отозвать заявку, только пока на неё никто не откликнулся.
  После первого отклика — отозвать нельзя; зависшую заявку удаляет администратор вручную (см. 1.7).
- **Защита от манипуляции ценами:** исполнитель ни в ленте, ни на странице заявки не должен иметь
  возможности узнать, есть ли уже отклики на заявку (кроме своего собственного) — знание о наличии/
  отсутствии конкурентов искажает предложение цены/срока в обе стороны. См. новый инвариант ниже
  и техническую реализацию в `architecture.md` §4.3.
- **Кабинеты (навигация по статусам заявок и откликов):**
  - Заказчик — «Мои заявки» = **открытые** (статусы `open`/`under_review`/`awarded`/
    `result_submitted` — заявка ещё в процессе) + **архив** (статус `accepted` — закрытые сделки).
  - Исполнитель — **два раздельных раздела**: «Мои отклики» (все заявки, на которые откликался,
    со статусом самого отклика: ожидает рассмотрения / рассматривают / выбран / не выбран) и
    «В работе и выполненные» (только заявки, которые исполнитель выиграл — `awarded` и далее,
    независимо от статусов его откликов на другие заявки).
**Приёмка:** полный путь заявка → отклик → рассмотрение → раскрытие контакта → выбор из
рассмотренных проходит от начала до конца; выбор нерассмотренного отклика невозможен; отзыв
заявки блокируется после первого отклика; исполнитель не может определить по ленте/API-ответу,
есть ли на заявку другие отклики; заказчик видит открытые заявки отдельно от архива; исполнитель
видит «мои отклики» и «в работе и выполненные» как два раздельных раздела, не один список.

### 1.5. Двусторонние вехи завершения
- После выбора: исполнитель отмечает «результат сдан» (с загрузкой отчёта/файлов).
- Заказчику прилетает «принять / вернуть на доработку». Статус «принято» ставит только заказчик.
- Заявка закрывается только после подтверждения заказчиком.
**Приёмка:** исполнитель не может закрыть сделку сам; пока заказчик не принял — заявка открыта, отзыв недоступен.

### 1.6. Геомодуль (база)
- Загрузка KML / GeoJSON, парсинг, превью на карте. SHP — только zip или не поддерживать.
**Приёмка:** загруженный KML/GeoJSON корректно отображается на карте заявки/объекта.

### 1.7. Базовая админка (Django Admin)
- Управление пользователями, заявками, верификацией, жалобами.
- Администратор может **удалить любую заявку** — единственный ручной рычаг для зависшей заявки
  или по просьбе заказчика, поскольку сам заказчик не может отозвать заявку после первого
  отклика (см. 1.4).
**Приёмка:** оператор управляет ключевыми сущностями через Django Admin; администратор может
удалить любую заявку.

### 1.8. Трёхъязычие и темы (заложить с начала)
- Все строки через i18n (kk/ru/en). Тёмная/светлая темы через токены.
**Приёмка:** интерфейс переключается между тремя языками и двумя темами без сломанной вёрстки.

### 1.9. Доменные события (заложить с начала)
- Эмиссия событий из §7 CLAUDE.md в момент действий.
**Приёмка:** ключевые действия порождают события (видно в логе/таблице), даже если потребитель пока один.

### 1.10. Репутация (перенесено из Волны 2 — теперь MVP)
- После закрытия заявки заказчик может (по желанию) оставить отзыв: звёзды (1–5) +
  теги-галочки из расширяемого справочника (только положительные, по аналогии с такси) +
  необязательный текстовый комментарий.
- Полностью опционально — заявка закрывается без отзыва, если заказчик не оставил его.
- Доступно только после статуса «закрыта».
**Приёмка:** заказчик может оставить отзыв только после закрытия сделки; отзыв необязателен;
список тегов редактируется через справочник (админку), без изменения кода.

### 1.11. Почта и уведомления (перенесено из Волны 2 — теперь MVP)
- Подтверждение почты при регистрации — обязательно.
- Сброс пароля по почте.
- Исполнителю: «вас рассматривают» (при нажатии «рассмотреть»), «вас выбрали».
- Исполнителю: результат верификации (одобрено/отказ с причиной) — на почту И в кабинет.
- Заказчику: письмо только при ПЕРВОМ отклике на его заявку (не при каждом последующем).
- **Гейт подтверждения почты — мягкая блокировка:** вход разрешён сразу после регистрации
  (`is_active` не меняется), но создание заявки (заказчик) и отклик (исполнитель) заблокированы,
  пока почта не подтверждена. Пользователь видит баннер «подтвердите почту» с возможностью
  переотправить письмо; как только подтвердил — ограничения снимаются немедленно.
- Существующие на момент внедрения пользователи считаются подтверждёнными (грандфазеринг) —
  ограничение действует только для новых регистраций.
**Приёмка:** письма реально отправляются (не только console-backend); эмиссия — асинхронно
(Celery), не блокирует основной запрос; неподтверждённый пользователь не может создать заявку
или откликнуться, но может войти и просматривать интерфейс; существующие пользователи не
заблокированы задним числом.

### 1.12. AI-генерация ТЗ (упрощённая версия — теперь MVP)
- При создании заявки: есть готовое ТЗ → прикладывает файл (как сейчас); нет — заполняет
  короткий опросник → система генерирует черновик по шаблонному промпту.
- НЕ RAG, черновик не сверяется с нормами РК (полноценная версия — Волна 4+).
- Обязательный дисклеймер: сгенерировано ИИ, платформа не отвечает за полноту/соответствие
  нормам, заказчик должен проверить и дополнить, при необходимости — уточнить с исполнителем.
**Приёмка:** черновик генерируется и заменяет ручной ввод ТЗ; дисклеймер показан перед
использованием черновика; персональные данные не попадают в промпт.

## ВОЛНА 2 — Доверие, репутация, удержание

Цель: сделать платформу местом, где ценно оставаться, а не только знакомиться.

### 2.1. Жалобы (репутационная гигиена)
- Кнопка «пожаловаться на исполнителя» (заказчик описывает проблему). Жалобы видны оператору,
  паттерн отслеживается. (Отзывы/рейтинг исполнителя — теперь 1.10 в Волне 1.)
**Приёмка:** жалоба создаётся и видна в админке.

### 2.2. Договоры в системе
- Шаблон договора об оказании услуг, заполняется внутри платформы, привязан к объекту/сделке.
- Хранение договоров и актов по объекту.
**Приёмка:** стороны формируют договор в системе; он сохраняется в истории объекта. *(До продакшена — юридическая сверка шаблона под РК.)*

### 2.3. Кабинет заказчика (минимум ретеншна)
- Карточки объектов с историей заказов, файлов и договоров под каждым.
- Добавление исполнителя в избранное (со статусом допуска и рейтингом).
**Приёмка:** заказчик видит свои объекты с полной историей и список избранных исполнителей.

### 2.4. Уведомления исполнителю
- Уведомление о новых заявках в нише/городе исполнителя.
**Приёмка:** исполнитель получает уведомление при появлении релевантной заявки.
(Базовая почтовая инфраструктура и остальные письма — теперь 1.11 в Волне 1; здесь остаётся
только специфичное «новая заявка в нише/городе», которое пока не в MVP.)

### 2.5. Операторская панель (read model)
- Метрики: fill rate, время до первого отклика, конверсия заявка→сделка, в разрезе ниша × город.
- На старте — отчёты/вьюхи в Django Admin поверх событий.
**Приёмка:** оператор видит fill rate и ликвидность по сегментам; может определить пустые ниши.

## ВОЛНА 3 — Монетизация и ускорение ретеншна

Включается, когда операторская панель показывает: fill rate стабильно выше порога И достаточно активных заявок в месяц.

### 3.1. Биллинг и подписка
- Подписка для исполнителей, интеграция с платёжным провайдером РК.
- **Feature-toggle монетизации** (до этого момента выключен).
- Грэндфэзеринг ранних исполнителей (сниженная цена навсегда для первых N).
**Приёмка:** монетизация включается настройкой; ранние исполнители получают льготную цену; подписка-перед-просмотром заявок работает.

### 3.2. Ускорение ретеншна заказчика
- «Повторить заказ» одной кнопкой (прошлая заявка как шаблон).
- Сохранённые шаблоны типовых работ («топосъёмка под ИЖС», «геология под фундамент»).
**Приёмка:** заказчик повторяет заказ из истории/шаблона за меньшее число шагов, чем создание с нуля.

## ВОЛНА 4+ — Отложенное (по подтверждению спроса)

- **ИИ-генерация ТЗ — полная версия (RAG):** корпус норм РК в pgvector, обязательная
  цитируемость источников по каждому пункту. Упрощённый черновик без RAG — уже в MVP (1.12).
  Спрос на полную версию проверить вручную («консьерж-MVP») до постройки.
- **Тендер / ставки + анти-демпинг:** когда на заявку реально приходит много откликов и есть конкуренция.
- **Автоматизация верификации:** автосверка лицензий/аттестатов по госреестрам РК (elicense.kz, реестр ИТР).
- **SMS-подтверждение телефона при регистрации:** код через SMS-шлюз РК, с rate limiting и защитой от абьюза.
- **Автопроверка/автозаполнение по БИН (юрлица):** сервис `identity_lookup` поверх ГБД ЮЛ / коммерческого агрегатора / локального дампа; данные на подтверждение пользователю.
- **Автозаполнение по ИИН (физлица) через Egov/ЭЦП:** возможно только с авторизацией человека; увязать с Egov-интеграцией (заодно закрывает верификацию личности).
- **Мобильное приложение:** подключается к существующему API.
- **Слот будущих модулей:** аренда оборудования, инструменты проектировщика.

## Сводка анти-целей (что НЕ делаем и почему)

| Не делаем в MVP | Причина |
|---|---|
| Эскроу / расчёты внутри | Расчёты идут мимо платформы; подписка защищает от обхода |
| Тендер + анти-демпинг | Имеет смысл только при конкуренции за заявку, которой пока нет |
| ИИ-генерация ТЗ (полная, RAG) | Дорого и рискованно без проверенного спроса; упрощённый черновик без RAG — уже в MVP |
| Арбитраж споров | Платформа не сторона сделки; берём только репутационную гигиену |
| Монетизацию на старте | Сначала ликвидность, потом деньги, иначе исполнители уйдут |

## Ключевые продуктовые инварианты (не нарушать на любом этапе)

1. Отзыв возможен **только** после подтверждённой заказчиком сделки.
2. Статус «принято» ставит **только** заказчик.
3. Объект (`site`) — первоклассная сущность; заявки/сделки/договоры/файлы привязаны к нему.
4. Монетизация — за feature-toggle, выключена до достижения порога ликвидности.
5. Верификация исполнителя — опциональный процесс: неверифицированный откликается без
   ограничений, статус верификации виден заказчику в каждом отклике.
6. Выбор исполнителя возможен только из рассмотренных заказчиком откликов (минимум один
   рассмотренный обязателен).
7. Заявку можно отозвать только пока на неё нет ни одного отклика; после первого отклика —
   только через администратора.
8. Отзыв и оценка — полностью опциональны, доступны только после статуса «закрыта».
9. До закрытия сделки исполнитель не может узнать, есть ли отклики на заявку (кроме
   собственного) — ни через статус, ни через `updated_at`, ни любым другим полем/каналом.
   Защита от манипуляции ценами: знание о наличии/отсутствии конкурентов искажает
   предложение цены/срока в обе стороны. После перехода заявки в `accepted` торг окончен,
   и факт победы перестаёт быть скрытым: отзыв заказчика публикуется с его именем и
   привязкой к исполнителю (PRODUCT_SPEC 1.10).
