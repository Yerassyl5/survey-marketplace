## 3. Что отложено на потом

**Пред-деплойные задачи (блокируют прод, не разработку):**
- Gunicorn вместо `runserver`, `next build + next start` вместо dev
- Маршрутизация Coolify/Traefik: `/api` → backend, остальное → frontend, HTTPS
- `DEBUG=0`, `ALLOWED_HOSTS`, `collectstatic`
- Замена всех `change-me` паролей (`MINIO_ROOT_PASSWORD`, `POSTGRES_PASSWORD`, `DJANGO_SECRET_KEY`) — **критично, сейчас дыра безопасности**
- Регулярный бэкап БД (pg_dump → MinIO/внешнее, политика хранения, проверка восстановления)

**Техдолги из Волны 1:**
- Жёсткая блокировка неверифицированных исполнителей (`REQUIRE_VERIFIED_TO_BID=True`) — одна строка, но сознательно отложена

**Волна 2 (доверие и ретеншн):**

## 4. Что логично делать следующим и почему

**Ближайшие два шага:**

1. **Сверить и закрыть 1.5** — 10 минут: проверить, действительно ли submit/accept/return
   работают с загрузкой файлов, и проставить [x] там где реализовано. Это разблокирует
   чёткую картину — сколько реально осталось в бэкенде Волны 1.

2. **1.6 Геомодуль (KML/GeoJSON)** — следующая цельная backend-задача, логически связана
   с 1.3 (объекты) и 1.4 (заявки с геометрией). Это последний крупный backend-кусок
   Волны 1, после которого backend MVP будет feature-complete.

**После этого — фронтенд.** Весь backend 1.1–1.6 будет готов, но без UI пользователь
не может пройти путь end-to-end. Формы регистрации, лента заявок, страница объекта
с картой — это то, что превратит MVP в проверяемый продукт. Именно фронтенд сейчас
является самым большим разрывом между тем, что есть, и тем, что можно показать
живым пользователям.
