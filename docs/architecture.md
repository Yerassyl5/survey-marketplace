# Архитектура системы

> Полное описание **как устроена система**. Читать перед архитектурными задачами.
> Операционные правила — в корневом `CLAUDE.md`. Что и в каком порядке строить — в `PRODUCT_SPEC.md`.

---

## 1. Архитектурный стиль — модульный монолит

**Это модульный монолит, НЕ микросервисы.** Одно разворачиваемое Django-приложение, разбитое на bounded contexts (модули) с чёткими границами.

Правила модульности:
- Каждый модуль — отдельное Django-приложение со своими моделями, своим публичным API (сервисный слой) и минимальной связностью.
- Модули **не** ходят друг к другу по HTTP. Общение между модулями — вызовы публичных интерфейсов в коде + доменные события.
- Внешняя граница (клиент ↔ бэкенд) = REST API. Внутренняя граница (модуль ↔ модуль) = публичные сервисы модулей в памяти.
- Принцип «monolith first, extract later»: когда модуль упрётся в потолок нагрузки (первый кандидат — ИИ-генерация ТЗ), его вырезают в отдельный сервис по уже проведённой границе.
- Тяжёлые/долгие задачи уходят в **Celery-воркеры** (отдельно масштабируемый процесс): парсинг гео-файлов, рассылки, в будущем — генерация ТЗ.

## 2. Технологический стек

Всё open-source, платим только за ресурсы и внешние услуги (сервер, LLM-токены позже, домен, платёжный провайдер, SMS-шлюз).

- **Backend:** Python, Django, Django REST Framework (DRF)
- **БД:** PostgreSQL — одна база на всё:
  - реляционные данные (заявки, отклики, пользователи)
  - **PostGIS** — геоданные
  - **pgvector** — векторный поиск для ИИ (позже, для RAG)
- **Кэш / брокер очереди:** Redis
- **Очередь задач:** Celery (воркеры + beat)
- **Объектное хранилище:** MinIO (S3-совместимое, self-hosted). В БД — только ссылки на файлы.
- **Frontend:** Next.js (App Router) на React, SSR. Карта на MapLibre (тайлы OSM на старте). Отдельное самостоятельное приложение, общается с бэкендом только по REST API (Django — чистый API на DRF, без серверных шаблонов).
- **Контейнеризация:** Docker, docker-compose
- **Деплой:** Coolify на выделенной Ubuntu LTS VM (Hoster.kz). Образы собираются в GitHub Actions, Coolify тянет готовые образы.
- **Админка MVP:** Django Admin

**Важно по инфраструктуре:** стандартный PostgreSQL-шаблон Coolify НЕ содержит PostGIS и pgvector — нужен кастомный Dockerfile для образа Postgres с обоими расширениями. Django-образ требует системные GDAL/GEOS/PROJ.

## 3. Требования к платформе (сквозные)

- **Трёхъязычие с первого дня:** казахский (кириллица), русский, английский. Django i18n (для серверных сообщений/админки) + **next-intl** на фронтенде (App Router, маршрутизация с языковым префиксом `/kk`, `/ru`, `/en`). Все пользовательские строки — через систему переводов, никакого хардкода текста. ВАЖНО: встроенная i18n-конфигурация из `next.config.js` (Pages Router) НЕ работает в App Router — использовать подход next-intl под App Router, не смешивать со старыми гайдами под Pages Router.
- **Тёмная и светлая темы:** через CSS-семантические токены, не хардкод цветов.
- **API-first:** бэкенд клиент-агностичен (Django/DRF отдаёт только REST API). Next.js-фронтенд — лишь один из клиентов; мобильное приложение в будущем подключится к тому же API. Не закладывать клиент-специфичные допущения в API. Смена фронтенд-фреймворка не должна затрагивать бэкенд.
- **Резидентность данных:** все данные хранятся на VM в Казахстане. Границу пересекает только вызов LLM API (позже), причём из промптов вырезаются персональные данные.

## 4. Модули (bounded contexts)

### 4.1. Доступ и верификация (`accounts`)
Аутентификация, два типа участника (заказчик / исполнитель), роли.

**Регистрация (MVP — ручной ввод):**
- При регистрации пользователь сначала выбирает тип лица: юридическое или физическое.
- Физлицо вводит вручную: ФИО, ИИН, почта, телефон.
- Юрлицо вводит вручную: ФИО (контактное лицо / руководитель), БИН организации, почта, телефон.
- Поля БИН/ИИН на старте — просто сохраняются как введённые, БЕЗ автоматической проверки и автозаполнения.
- SMS-подтверждение телефона на старте НЕ делаем (докрутим позже).

**Отложено (заложить абстракции, не реализовывать в MVP):**
- *SMS-подтверждение телефона:* код с TTL в Redis, отправка через SMS-шлюз РК (платный, абстрагировать в `notifications`). Обязательно rate limiting на отправку и лимит попыток ввода — защита от абьюза SMS-бюджета.
- *Автопроверка/автозаполнение по БИН:* данные юрлиц/ИП — открытые (ГБД ЮЛ Минюста, data.egov.kz API, либо коммерческие агрегаторы, либо локальный дамп). Абстрагировать за сервисом `identity_lookup` с единым интерфейсом «данные по БИН», чтобы менять источник без переписывания регистрации. Автозаполненные данные показывать пользователю на подтверждение, не подставлять молча.
- *Автозаполнение по ИИН (физлица):* ФИО по ИИН — персональные данные, в открытом доступе НЕТ. Возможно только через авторизацию человека (ЭЦП/Egov) с его согласием. Поэтому для физлиц автозаполнение из госисточника невозможно без Egov-интеграции — либо ручной ввод, либо путь через ЭЦП (увязать с будущей Egov-интеграцией, которая заодно закрывает верификацию личности).

**Верификация исполнителя — критична.** Исполнитель при регистрации загружает разрешительные документы (лицензия на изыскательскую деятельность, квалификационные аттестаты ИТР). На старте — ручная проверка модератором (сверка с госреестрами РК глазами), статус «верифицирован» ставится вручную.
Поля модели исполнителя закладывать сразу, даже при ручной проверке (для будущей автосверки без миграций): `license_number`, `attestation_number`, `license_expiry`, `verification_status`, `verification_method` (manual/auto).
На будущее — интеграция с ЭЦП/Egov и автосверка по реестрам elicense.kz и реестру аттестованных ИТР.

### 4.2. Объекты (`sites`) — ПЕРВОКЛАССНАЯ СУЩНОСТЬ
**Земельный участок / объект** — это сущность, к которой привязываются заявки, сделки, договоры и файлы. Заказчик мыслит участками, а не транзакциями.
Это фундамент ретеншн-механики (кабинет заказчика = проекция данных, сгруппированная по объекту). **Заложить с самого начала**, иначе болезненная миграция, когда заявки уже накопятся без привязки к объекту.
Поля: адрес, точка/контур на карте (PostGIS), кадастровый номер, владелец-заказчик. Связь: один объект → много заявок/сделок/договоров/файлов.

### 4.3. Маркетплейс (`marketplace`)
Заявки, отклики, матчинг, выбор исполнителя. Заявка привязана к объекту (`sites`).
**Двусторонние вехи завершения (критично):**
- Заявка НЕ закрывается, пока заказчик не подтвердил выполнение работы.
- Оценку/отзыв исполнителю нельзя оставить, пока заявка не завершена подтверждением заказчика.
- Статус «принято» ставит ТОЛЬКО заказчик. Исполнитель не может в одностороннем порядке закрыть сделку.
- Минимум две двусторонние точки: «исполнитель сдал результат» → «заказчик принял / вернул на доработку».
Тендер/ставки и анти-демпинг — НЕ в MVP (см. PRODUCT_SPEC). На старте — простой «отклик + контакт».

### 4.4. Договоры (`contracts`)
Составление договора об оказании услуг **внутри платформы**. Удерживающая ценность для заказчика + легальный след сделки. Привязан к объекту и сделке. Хранение договоров и актов по объекту.
*Требует сверки юридической корректности шаблона под законодательство РК перед продакшеном.*

### 4.5. Репутация (`reputation`)
Отзывы и рейтинг исполнителя — **только после завершённой подтверждённой сделки** (завязано на вехи из `marketplace`, чтобы не накручивали).
Кнопка «пожаловаться на исполнителя» (заказчик описывает проблему). Цель — НЕ арбитраж сделки (платформа не вмешивается в споры сторон), а репутационная гигиена: видеть паттерн (например, 5 жалоб на одного → снятие исполнителя), защищать сеть и репутацию платформы.

### 4.6. Геоданные (`geo`)
Парсинг и хранение KML / GeoJSON. PostGIS. Превью на карте (MapLibre). Привязка геометрии к объектам и заявкам.
**Shapefile:** формат многофайловый — принимать только zip-архивом, либо не поддерживать SHP вовсе на старте.

### 4.7. Уведомления (`notifications`)
Канал уведомлений абстрагирован в отдельный модуль (на будущее — push для мобилки без переделки бэкенда).
Ключевое на старте: уведомлять исполнителя о новых заявках в его нише/городе (удержание исполнителя в маркетплейсе, пока заказчиков мало).

### 4.8. Биллинг (`billing`)
Подписка для исполнителей. **Feature-toggle монетизации с самого начала** — включение подписки это настройка, а не переписывание кода.
На старте всё бесплатно. Триггер включения: fill rate стабильно выше порога И достаточно активных заявок в месяц (подписка окупается ценностью). Предусмотреть грэндфэзеринг ранних исполнителей (сниженная цена навсегда для первых N).
Интеграция с платёжным провайдером РК.

### 4.9. Операторская аналитика (`analytics`) — READ MODEL, не bounded context
Аналитический слой (проекция чтения) поверх **доменных событий** остальных модулей. Ничего не пишет в чужие модули — только слушает и агрегирует.
Метрики ликвидности: fill rate (доля заявок с ≥1 откликом), время до первого отклика, конверсия заявка→сделка. Всё в разрезе **ниша × город** — видеть, какие сегменты живые, какие пустые.
Реализация: на старте — SQL-вьюхи / отчёты в Django Admin (быстро). Эмиссию доменных событий закладывать сразу, чтобы потом переключить на журнал событий без переписывания модулей.

### 4.10. Кабинет заказчика (ретеншн) — ПРОЕКЦИЯ, не новый модуль
Собирается из `sites` + `marketplace` + `contracts` + `geo`. Группировка по объекту.
Содержит: карточки объектов с историей заказов под каждым, сданные файлы/отчёты, договоры; «повторить заказ» (прошлая заявка как шаблон); сохранённые шаблоны типовых работ; избранные исполнители (с рейтингом и статусом допуска).
На старте — минимум: группировка заявок и файлов по объекту + добавление исполнителя в избранное. «Повторить заказ» и шаблоны — позже.

### 4.11. Генерация ТЗ на ИИ (`tz_ai`) — ОТЛОЖЕНО
НЕ в MVP. Подход — **RAG (Retrieval-Augmented Generation)**, НЕ fine-tune.
Корпус нормативки РК (СН РК, СП РК, ГОСТы, источники Adilet) индексируется в pgvector. Каждый сгенерированный фрагмент ТЗ обязан цитировать источник через метаданные (выдуманная ссылка на пункт СНИП = юридическая проблема для заказчика). Целостность ссылок критична.
Спрос проверяется вручную («консьерж-MVP») до постройки автоматизации.

### 4.12. Слот будущих модулей
Аренда оборудования для геодезистов, инструменты для проектировщиков. Каждый подключается как новый bounded context, не трогая существующий код.

## 5. Доменные события (фундамент аналитики и связности)

Каждый модуль эмитит события в момент значимых действий. Закладывать с начала, даже если потребитель пока один.

Примеры:
- `SiteCreated`
- `RequestCreated` (с нишей, городом, объектом)
- `BidPlaced` (привязка к заявке, метка времени)
- `RequestAwarded`
- `ResultSubmitted`
- `RequestAccepted` (только заказчиком)
- `ContractorDocumentsSubmitted` (исполнитель загрузил/обновил сканы лицензии/аттестата — триггер для уведомления модератора)
- `DealCompleted`
- `ReviewLeft`
- `ComplaintFiled`
- `SubscriptionActivated`

На этих событиях строятся: операторская аналитика (§4.9), уведомления (§4.7), репутация (§4.5).

**Аудит:** доменные события нужно **персистировать в append-only таблицу-журнал** (`AuditLog` — автор, тип события, payload, timestamp). Журнал используется для разбора споров и операционной безопасности. Это не то же самое, что аналитика (§4.9): журнал хранит сырые события, аналитика агрегирует поверх них. Реализуется отдельным подписчиком на события, не меняя эмиттеры.

## 6. Безопасность (архитектурный уровень)

- REST API: аутентификация, авторизация по ролям на каждом эндпоинте.
- Валидация и санитизация всех загружаемых файлов (гео-файлы, документы, сканы лицензий).
- Персональные данные не уходят во внешние API (актуально для будущего LLM-слоя).
- Файлы — в MinIO, не в БД; в БД только ссылки + права доступа.
- Секреты — через переменные окружения, никогда в коде или гите.

## 7. Чего НЕ делать (явные архитектурные анти-цели)

- НЕ строить эскроу / расчёты внутри платформы (расчёты идут мимо).
- НЕ строить тендер/ставки и анти-демпинг в MVP.
- НЕ строить ИИ-генерацию ТЗ в MVP (RAG позже).
- НЕ брать на себя арбитраж споров сторон (только репутационная гигиена).
- НЕ включать монетизацию на старте (feature-toggle выключен).
- НЕ делать модули, общающиеся по HTTP (это монолит, а не микросервисы).

## 8. Ключевые инварианты (дублируются в CLAUDE.md, не нарушать)

1. Отзыв возможен только после подтверждённой заказчиком сделки.
2. Статус «принято» ставит только заказчик.
3. Объект (`site`) — первоклассная сущность; заявки/сделки/договоры/файлы привязаны к нему.
4. Монетизация — за feature-toggle, выключена до достижения порога ликвидности.
5. Исполнитель проходит верификацию допусков (на старте ручную) до возможности откликаться.
6. Все пользовательские строки — через i18n (kk/ru/en).
7. Модули не общаются по HTTP; только публичные сервисы в памяти + доменные события.

## 9. Веб-сервер и деплой (production)

> Статус: описание целевого продакшен-стека. На момент написания в каркасе используются только dev-серверы — переход на этот стек — задача будущей волны деплоя (см. `docs/progress.md`).

**Dev vs prod:**
- Сейчас для локальной разработки используются dev-серверы: Django `runserver` на 8000, Next.js dev-сервер (`next dev`) на 3000. Это нормально для разработки, но **не предназначено для продакшена** (нет многопроцессности, отсутствуют прод-оптимизации, dev-сервер Django явно предупреждает о небезопасности для прод-нагрузки).

**Backend (Django):**
- Прод-цепочка запроса: реверс-прокси → **Gunicorn** (WSGI-сервер, несколько воркеров) → Django.
- В команде запуска backend-контейнера в проде `runserver` заменяется на запуск через `gunicorn`.

**Reverse proxy / HTTPS:**
- Терминацию HTTPS/SSL и обратное проксирование берёт на себя **Coolify** (под капотом — Traefik либо Caddy, в зависимости от конфигурации Coolify).
- Маршрутизация: `/api` → backend (Gunicorn/Django), всё остальное → frontend (Next.js).
- Отдельный ручной Nginx-конфиг по умолчанию не нужен — маршрутизацию обеспечивает встроенный прокси Coolify, но её необходимо явно проверить при настройке деплоя (правильность путей `/api`, корректная передача заголовков, HTTPS-редирект).

**Frontend (Next.js):**
- В проде Next.js запускается собственным production-режимом: `next build` + `next start`, а не dev-сервером (`next dev`).

**Статика и медиа:**
- Django-статика собирается через `collectstatic`.
- Медиафайлы раздаются либо через реверс-прокси, либо напрямую из MinIO (S3-совместимое хранилище, уже используется в проекте под файлы) — конкретный вариант фиксируется на этапе деплоя.

**Django prod-настройки:**
- `DEBUG=0`.
- Корректный (непустой, ограниченный реальными доменами) `ALLOWED_HOSTS`.
- Все секреты — только из переменных окружения, никогда в коде или дефолтных значениях настроек.

**Бэкапы БД:**
- Docker volume защищает данные Postgres от пересборки/удаления контейнера, но НЕ защищает от отказа диска или сервера — это не резервная копия.
- В продакшене обязателен регулярный `pg_dump` по расписанию с выгрузкой дампов в отдельное хранилище (MinIO либо внешнее), физически отделённое от диска БД.
