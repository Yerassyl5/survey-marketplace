## Этап 5 — токены сброса пароля, сервисный слой

**Ключевое архитектурное решение, пересмотренное дважды в процессе.** Исходный план пользователя
(`uid`+`token` в двух query-параметрах, `PasswordResetTokenGenerator` напрямую) заменён на
двухслойную схему: внутренний слой — `django.contrib.auth.tokens.PasswordResetTokenGenerator`
(хэширует `user.password` И `user.last_login`, значит смена пароля сама инвалидирует
неиспользованную ссылку — то самое свойство, которого не хватало и из-за отсутствия которого
этот генератор был **отвергнут** для подтверждения почты, где та же привязка была дефектом);
внешний слой — `signing.dumps`/`loads` вокруг внутреннего токена, тот же приём, что
`EMAIL_VERIFICATION_SALT`, даёт различимые исключения (`SignatureExpired`/`BadSignature`) через
полностью публичный API, без обращения к приватным методам генератора. Итог — ссылка с ОДНИМ
параметром (`?token=...`), не парой `uid`+`token`.

**Три исключения, не два** — `PasswordResetTokenExpired`/`PasswordResetTokenInvalid`/
`PasswordResetTokenAlreadyUsed`. Третье добавлено по возражению пользователя на мой
первоначальный план «один общий текст на все ошибки»: человек, открывший письмо ПОСЛЕ того, как
уже сменил пароль другим путём, не должен видеть пугающее «ссылка недействительна». Различение
безопасно формулировать конкретно («пароль уже был изменён») — проверено фактом, не
предположением: `api_settings.UPDATE_LAST_LOGIN == False` в проекте, единственная практическая
причина несовпадения здесь это реальная смена пароля, не вход с другого устройства.

`blacklist_all_refresh_tokens(user)` вынесен из `ChangePasswordView` в `accounts/services.py` —
второе место использования (сброс по ссылке) оправдало вынос. TTL — 1 час (короче, чем 3 суток у
email-verification — самый чувствительный токен в системе).

## Этап 6 — эндпоинты, письмо, троттлинг

`POST /accounts/request-password-reset/` — единый ответ независимо от существования email
(`filter().first()` занимает одинаковое время для обоих случаев, в отличие от `LoginSerializer` с
дорогим bcrypt — искусственный тайминг-паритет здесь не нужен). `POST /accounts/reset-password-confirm/`
— три кода ошибки. Троттлинг по IP (`ScopedRateThrottle`, `5/hour`/`10/hour`).

Три новых события: `PasswordResetRequested` (только если email найден — иначе журнал стал бы
каналом перебора адресов), `PasswordResetCompleted`, `PasswordChanged`. Payload каждого — строго
`{"user_id": int}`.

**Находка, не предположение:** утечка троттлинг-счётчика между тестами — троттлинг здесь по IP
(не по `user.pk`, как `resend-verification`), все вызовы Django test client используют один и тот
же фиктивный IP, счётчик `LocMemCache` общий для всех тестов в одном прогоне. Разобрано до
причины (не подогнано под ожидаемый результат), `cache.clear()` добавлен в `setUp()` четырёх
классов.

Живая проверка — реальный HTTP + Mailpit, полный цикл до смены пароля и входа новым.

## Этап 7 — видимость для оператора: история входов, подтверждение почты, привязка к оператору

**`UserLoggedIn`, не `last_login`.** Найдено фактом: `last_login` не обновляется в проекте вообще
(`UPDATE_LAST_LOGIN` не задан явно, дефолт библиотеки `False`, единственный вызов
`update_last_login()` стоял за этим же флагом и не вызывался никогда — мёртвая ветка, удалена).
**Включать `UPDATE_LAST_LOGIN` нельзя** — сломало бы самоинвалидацию токена сброса (этап 5).
Вместо `last_login` — событие `UserLoggedIn(user_id)`, только на `POST /login/`, не на
`token/refresh/` (автоматическое продление сессии каждые 15 минут — не действие человека).

**`changed_by_user_id`** — в новом `EmailVerificationChangedByAdmin` и, задним числом, в уже
существующем `ContractorVerificationDecided` (этап 2 блока 1.11). Отличие от отклонённого в
этапе 1 блока 1.11 общего поля «автор» на `AuditLog`: там эвристика «угадать актёра по одному из
id» врала бы для части событий, здесь актёр — факт из `request.user` в `save_model()`, не
предположение. Доказано тестом с двумя разными модераторами.

`is_email_verified` в `UserAdmin` — `list_display`/`list_filter`/`list_editable`, редактируемо
(реальный путь поддержки, когда письмо не дошло по вине провайдера получателя), ручное изменение
публикует событие только при реальном переходе. `phone`/`person_type` убраны из списка
(не влезали), `person_type` остался в фильтре.

**Колонка «Последний вход» — bulk-запрос, не N+1**, но с находкой по пути: наивный
`self._last_logins` на инстансе `UserAdmin` был бы **небезопасен** — `ModelAdmin` создаётся один
раз и переиспользуется между запросами (синглтон), `runserver`/Gunicorn с несколькими
threads/workers обслуживают запросы конкурентно, два оператора затирали бы словарь друг друга
посреди рендера. Решение — `contextvars.ContextVar` (тот же примитив, которым Django/DRF
пользуются для per-request состояния). **Это третий случай в проекте из серии «работает в dev,
молча ломается в проде»** — после `EMAIL_TIMEOUT` (этап 0 блока 1.11) и `CACHES`/`LocMemCache`
(этап 3 блока 1.11); общий знаменатель всех трёх — dev-конфигурация с одним процессом без
тредов/воркеров, где race condition никогда не проявляется.

Расчёт идёт по ВСЕМ пользователям, не по видимой странице `ChangeList` — точный расчёт по
странице потребовал бы `Subquery` поверх `AuditLog` прямо в queryset `accounts/admin.py`, что
нарушает инвариант №12, либо своей реализации `ChangeList`. При нынешнем масштабе (десятки-сотни
пользователей) разницы нет; станет проблемой при тысячах-десятках тысяч (каждый рендер тянет
агрегат по всей таблице `AuditLog`). Сортировки по колонке нет и не будет дёшево — не поле и не
аннотация, Django Admin не умеет сортировать по вычисляемому методу.

Живая проверка — реальная сессия Django Admin (логин → CSRF → cookie jar), POST в
`list_editable`-формсет: включение флага → ровно одна запись с `changed_by_user_id`, совпадающим
с id вошедшего оператора; повторное сохранение тем же значением — без дубля.

## Этап 8 — фикс часового пояса

**Найдено живой проверкой (скриншот пользователя), не по коду.** Журнал `AuditLog` показывал вход
как 16:25 (местное время Астаны), колонка «Последний вход» для ТОГО ЖЕ события — 11:25. Разница
ровно 5 часов = UTC+5. Подтверждено фактом: `TIME_ZONE = "Asia/Almaty"`, `USE_TZ = True`.

Grep по всему `backend/` (`strftime|:%Y|:%d|:%H`) нашёл **ровно два** места ручного форматирования
дат во всём проекте: `UserAdmin.last_login_display` (этот же этап) и `AuditLog.__str__`
(существовал с этапа 1 блока 1.11 — тот же баг, просто был незаметен: влиял на заголовок
вкладки/хлебную крошку страницы записи, не на списочные колонки). Оба чинятся
`django.utils.timezone.localtime()` перед форматированием.

**API проверено отдельно и НЕ тронуто** — DRF `DateTimeField` отдаёт ISO-8601 с явным UTC-офсетом,
перевод в локальное время — дело фронта, не бэкенда.

Живая проверка на `bake@mail.ru` (id=31): заголовок страницы `AuditLog` было `11:25:03` → стало
`16:25:03`; колонка списка — `24.07.2026 16:25`; оба значения совпадают друг с другом и с
`created_at` записи.

не менялась.
