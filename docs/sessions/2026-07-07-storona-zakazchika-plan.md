# Сессия 2026-07-07 — Сторона заказчика: план утверждён, код не начат

## Решение сессии

Сторона исполнителя (лента, страница заявки, отклик) была закрыта в прошлой сессии. Начали
планирование стороны заказчика: форма создания заявки, переименование кабинета в «Мои заявки»,
доступ заказчика к общей ленте (обезличенно). Сессия закончилась (кончились токены у
пользователя) **до того, как был написан хоть один код** — рабочее дерево чистое, изменений в
этой сессии нет ни в одном файле. Единственная правка — документация (`docs/progress.md`, этот
файл).

## Ход планирования

Прочитал `docs/progress.md`, `docs/architecture.md`, `docs/design-system.md`. Исследовал текущий
код: `apps/sites/{models,serializers,views}.py`, `apps/geo/{views,serializers}.py`,
`apps/marketplace/{views,serializers}.py`, фронтенд (`FilterBar.tsx`, `SiteMap.tsx`,
`RequestRow.tsx`, `AppNav.tsx`, `(app)/layout.tsx`, `login/page.tsx`, `client.ts`).

Ключевой узкий момент, который нужно было прояснить перед кодом — как заказчик задаёт геометрию
объекта (`Site.geometry` обязательно в БД, `null=True`/`blank=True` нет). Обнаружено: это ДВА
независимых механизма на бэкенде — `SiteSerializer`/`SiteListCreateView.POST` принимает геометрию
напрямую в теле (GeoJSON `Feature`), а `SiteGeometryUploadView` отдельно обновляет геометрию уже
существующего объекта файлом (KML/GeoJSON). Предложил MVP-вариант (только точка на карте), но
пользователь настоял на обоих способах сразу (точка + файл) — контур участка важен исполнителю
для оценки объёма/цены, это ядро ценности платформы.

Также вызывался skill `ui-ux-pro-max` (`--domain ux`, `--stack react`) для гайдлайнов по
многошаговым формам/файловой загрузке — в итоге пользователь решил делать форму ОДНИМ экраном,
не визардом (рекомендация из скилла про «Step N of M» не пригодилась в таком виде).

## Утверждённый план (детали — в `docs/progress.md`, раздел «Сторона заказчика»)

Без новых миграций БД — только сериализаторы/вьюхи `marketplace` + фронтенд. **5 коммитов:**

1. **Backend** — `RequestSerializer.validate_site()` (защита от чужого `site_id` — найденная
   попутно дыра, формы раньше не было, эксплуатировать было некому); `?scope=feed` на
   `GET /api/marketplace/requests/` для роли `customer`; `RequestFeedForCustomerSerializer` +
   `RequestFeedForCustomerDetailSerializer` (обезличивание чужого `customer` через
   `to_representation`, без `has_bid`); `RequestDetailView` — заказчик может открыть чужую
   открытую заявку (обезличенно), `get_serializer()` выбирает сериализатор по владению.
2. **Frontend-рефакторинг** — `LocationCascadeSelect` вынесен из `FilterBar.tsx` в отдельный
   переиспользуемый компонент (`allowEmpty` true/false).
3. **Форма создания заявки (`/requests/new`)** — один экран (не визард), карточки-секции,
   порядок полей: тип работ → локация → описание → ТЗ-файл → примечание для исполнителей →
   (внизу) объект и геометрия (выбор существующего `Site` ИЛИ новый: адрес + вкладки «точка на
   карте» / «файл KML-GeoJSON»). Редирект после успеха — на `/requests/{id}`.
4. **«Мои заявки» (`/requests/my`)** — вместо `/dashboard`-заглушки, список своих заявок без
   клика на строку (просмотр откликов/award — следующий блок).
5. **Доступ заказчика к ленте** — фронтенд-часть (зависит от коммита №1): гвард пускает
   `customer`, `customer: CustomerBrief | null` в типах, колонка отклика скрыта для заказчика.

## Зафиксированный техдолг / будущее (в `docs/progress.md`)

**Автоопределение «Региона работ» по геометрии объекта** (запись в разделе 1.6, Геомодуль) —
идея пользователя: при простановке точки/контура система сама определяет область/район через
`ST_Contains` (PostGIS) и автозаполняет `LocationCascadeSelect`. Зависимость: справочнику `geo`
нужны геометрии границ регионов (сейчас только текстовые названия из КАТО) — отдельный массив
данных границ + поле геометрии на `Region`/`District`. Не делать сейчас, ручной выбор региона
остаётся в форме.
