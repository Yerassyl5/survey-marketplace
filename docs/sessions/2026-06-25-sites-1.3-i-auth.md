# Сессия 2026-06-25 — объекты sites (1.3) + JWT-аутентификация

## Что сделано

### 1.3 — Объекты (sites)
- Модель `Site` (`backend/apps/sites/models.py`): `owner` (FK на `User`, ограничен ролью customer), `address`, `geometry` — единое `GeometryField(srid=4326)` без жёсткого типа (допускает и точку, и контур-полигон одним полем вместо двух раздельных point/contour), `cadastral_number`.
- API на `djangorestframework-gis`: `GET/POST /api/sites/` (список/создание, только роль customer, `owner` подставляется из `request.user`), `GET /api/sites/<id>/` (только владелец, иначе 404). Update/delete не делали — не входили в критерии приёмки 1.3.
- Подвох с `djangorestframework-gis`: `GeoFeatureModelSerializer` не знает про базовый `GeometryField` без явного геометрического подтипа — без явного объявления поле уходит как `ModelField` и ломает сериализацию GeoJSON. Нужно объявлять `geometry = GeometryField()` (из `rest_framework_gis.fields`) явно в сериализаторе.
- Событие `SiteCreated` публикуется при создании. Регистрация в Django Admin через `GISModelAdmin` (карта-виджет).
- Задел на 1.4: `marketplace.Request` получит `FK(Site, related_name="requests")` без доп. миграций — `Site.owner`/`related_name="sites"` уже на месте.
- Смоук-тест через Django test client (`force_login`): создание/список/просмотр владельцем — 200/201, просмотр чужого объекта — 404, создание исполнителем/анонимом — 403, рендер Django Admin (список + форма) — 200.

### Аутентификация (добавлена внепланово — обнаружили пробел в спеке)
- Проверили `PRODUCT_SPEC.md`: логина не было как отдельного пункта, хотя 1.2/1.3 уже завязаны на `request.user`, а 1.4 без входа не сделать. Дополнили 1.1 пунктом про вход и критерием приёмки.
- JWT (`djangorestframework-simplejwt`) выбран вместо DRF Token — пара access/refresh подходит будущему мобильному приложению (Wave 4+) лучше статичного токена без TTL.
- Эндпоинты: `POST /api/accounts/login/` (email+password → `access`, `refresh`, `user_id`, `role`), `POST /api/accounts/token/refresh/` (ротация refresh с блеклистингом старого), `POST /api/accounts/logout/` (явный блеклистинг), `GET /api/accounts/me/` (текущий пользователь + `verification_status` для исполнителя).
- `rest_framework_simplejwt.token_blacklist` подключён (своя миграция, 2 таблицы). `DEFAULT_AUTHENTICATION_CLASSES` = `JWTAuthentication` + `SessionAuthentication` (сессии остаются для Django Admin). Access — 15 мин, refresh — 7 дней.
- Смоук-тест curl по полному циклу: регистрация → логин → `/me/` по токену → `/api/sites/` по токену (200/201) → 401 без токена → refresh ротирует и блеклистит старый (повторное использование старого refresh → 401 "blacklisted") → logout блеклистит refresh → неверный пароль → 401. Swagger/OpenAPI схема подхватила новые эндпоинты без ошибок.
- `backend/apps/accounts/requests.http` дополнен примерами login/me/refresh/logout.
