## Этап 1 — причина отклонения верификации + сброс статуса (коммит `666a130`)

Разведка перед кодом: грепом по всему backend найдено **ровно одно** сравнение
`verification_status` в продакшен-коде (`ContractorCanBid`, `== VERIFIED`, за выключенным
фиче-флагом `REQUIRE_VERIFIED_TO_BID`) — риск от новых состояний минимален с самого начала.

`ContractorProfile.rejection_reason` (TextField) — заполняется модератором в админке. Ключевое
решение: `ContractorDocumentUploadSerializer.update()` теперь **безусловно** сбрасывает
`verification_status → pending` и очищает причину при любой пересдаче сканов — включая
`verified → pending`, не только `rejected → pending`. Обоснование пользователя: без этого
верифицированный исполнитель мог бы молча подменить скан лицензии, оставшись «верифицирован»
перед заказчиком на непроверенных документах. Тест `test_reupload_resets_verified_to_pending`
покрывает именно этот случай явно.

Найден и починен побочный баг в процессе: `ContractorPublicSerializer`-подобные места, где
`portfolio_description`/`verification_status` читались напрямую с `user.contractor_profile` без
`getattr(..., None)` — риск 500 при рассинхроне создания `User`/`ContractorProfile` (не в одной
транзакции). Защищено везде тем же приёмом, что уже был в `ContractorBriefSerializer`.

Тесты: accounts 4/4 (новый файл — тестов на `accounts` не было вообще до этого этапа), полный
прогон 137/137. Миграция `0004_contractorprofile_rejection_reason` (схема).

## Этап 2 — эндпоинт профиля, смена пароля, валидация телефона (коммит `5509327`)

**Почему не раздули `/me/`:** `/me/` дёргается на каждой загрузке приложения из `AuthContext` —
добавлять туда `portfolio_description`/`rejection_reason`/сканы значило бы тянуть их всегда, а не
только на экране настроек. Завели отдельный `GET/PATCH /accounts/profile/` (`ProfileSerializer`).
Редактируется ТОЛЬКО `phone` (обе роли) и `portfolio_description` (только contractor) — всё
остальное `read_only_fields`.

**Смена пароля** — `POST /accounts/change-password/`, требует текущий пароль. Блеклистит **ВСЕ**
`OutstandingToken` пользователя (не только текущий refresh) — после смены пароля все устройства
требуют повторного входа. Стоило исследования: `SIMPLE_JWT` не инвалидирует токены сам по себе,
`token_blacklist` — ручной механизм, тот же паттерн, что уже в `LogoutView`, применённый ко ВСЕМ
токенам пользователя, не одному.

**Валидатор телефона** — факт до реализации: запрос по dev-БД показал 14 из 30 телефонов не
проходят предложенный мягкий regex, но ВСЕ 14 — тестовые заглушки вида `123` (1–3 цифры), ни
одного реального номера. Валидатор безопасен, введён как пример в ошибке («Введите номер
телефона, например +7 701 000 00 00»), не как жёсткое требование формата — принят и на
регистрацию (раньше проверялось только «поле не пустое»).

**Найденный и исправленный баг:** `portfolio_description` в `ProfileSerializer` молча исчезало из
ответа для ОБЕИХ ролей — поле объявлено голым `CharField` без `source`, а физически лежит на
`ContractorProfile`; DRF тихо пропускал его (`SkipField`, `required=False`). Починено через
`to_representation()` тем же приёмом, что уже в `ContractorRegistrationSerializer`.

Тесты: accounts 21/21, полный прогон 154/154.

## Этап 3 — публичные данные исполнителя + список отзывов (коммит `15e0683`)

**Главный архитектурный вопрос:** карточке нужны данные из ДВУХ модулей (`accounts` — имя/
верификация/«О себе», `reputation` — рейтинг/отзывы). Разобрали оба варианта: один агрегирующий
эндпоинт vs два отдельных. Выбраны **два отдельных** (`GET /accounts/contractors/{id}/` +
`GET /reputation/contractors/{id}/reviews/`) — карточка на фронте состоит из двух независимых
визуальных блоков, объединение потребовало бы новой связности между модулями ради экономии одного
HTTP-запроса. Инвариант №12 запрещает модулям HTTP-общение МЕЖДУ СОБОЙ на бэкенде — числа фронтовых
запросов он не касается.

Рейтинг — переиспользован `get_ratings_for_contractors([pk])` (уже существовал с блока
«Репутация»), без отдельной «single-id» функции — она была бы обёрткой без новой логики.

**404 — одинаково** для несуществующего id и id заказчика в ОБОИХ эндпоинтах: различать означало
бы дать перебором узнать, какие id заняты и кто из них заказчик. Правило «это исполнитель»
продублировано в двух модулях (`accounts` — фильтр queryset, `reputation` — явный `.exists()`) —
зафиксировано комментариями с указанием, что менять синхронно.

Тесты: accounts+reputation 61/61, полный прогон 164/164.

## Этап 4 — страница настроек `/ru/settings` + меню в шапке (коммит `5cb1cb0`)

Четыре НЕЗАВИСИМЫХ блока сохранения (телефон / пароль / «О себе» / документы), каждый со своей
кнопкой — единый сабмит был бы неверен (ошибка пароля не должна блокировать сохранение телефона).

**Находка при реализации:** `GET /accounts/profile/` не отдавал, загружены ли сканы вообще — без
этого «не загружал» и «на проверке» неразличимы. Добавлены `has_license_scan`/
`has_attestation_scan` (read-only, вычисляются из `bool(FileField)`).

**Вход в настройки** — сначала клик по имени в шапке вёл прямо на `/ru/settings`; по итогам
браузерной проверки заменено на выпадающее меню («Настройки»/«Выйти», написан с нуля — готового
dropdown-компонента в проекте не было). Клик вне меню и `Escape` закрывают его.

**Смена пароля — особый выход.** После успеха backend уже блеклистит ВСЕ токены — обычный
`useAuth().logout()` нельзя звать (сам шлёт `POST /accounts/logout/` с уже блеклистнутым refresh и
падает ошибкой). Решение: `clearTokens()` напрямую + явная кнопка «Войти» (без авто-редиректа по
таймеру — в проекте нет ни одного `setTimeout`-редиректа).

Live-фикс по фидбеку: состояние «документов ещё не загружал» сделано САМЫМ заметным из четырёх
(амбер-фон, явный призыв к действию) — это первое, что видит новый исполнитель.

Тесты: accounts 28/28, полный прогон 166/166.

## Задача 8 — новый статус `not_submitted` + причина в списке админки

Найдено при проверке этапа 4: `pending` означало одновременно «только зарегался, ничего не
подавал» и «подал документы, ждёт решения» — админ не мог отличить одно от другого в списке.

**Решение (реальное значение в `VerificationStatus`, не вычисляемая колонка):** риск оказался
низким — единственное сравнение в проекте (`== VERIFIED`) не задевается новым значением.
`NOT_SUBMITTED` — новый дефолт модели вместо `PENDING`. Data-миграция (`RunPython`, паттерн
`apps.get_model` + литеральные строки, как уже в `marketplace/0006_backfill_under_review`)
перевела pending-БЕЗ-единого-скана записи в `not_submitted` — проверено запросом до и после:
10 из 11 dev-записей сменили статус, 11-я (с одним сканом) осталась `pending` — частичная
загрузка уже требует внимания модератора, не «ничего не подано».

Причина отклонения — `list_editable` в списке админки (не усечённая read-only колонка, как у
`RequestAdmin.contractor_note_column` — редактируемость важнее компактности). Виджет ужат через
`formfield_for_dbfield` **по имени поля**, не `formfield_overrides` по типу `TextField` — на
модели есть ещё `portfolio_description` того же типа, тип-scoped override задел бы и его.

Фронт **сознательно не переписан** под новый статус (техдолг, не срочно) — `/ru/settings`
продолжает различать состояния через `has_license_scan`/`has_attestation_scan`.

Тесты: accounts 30/30, полный прогон 168/168.

## Этап 5 — публичная карточка `/ru/contractors/[id]` (коммиты `8db27ef`, `831248a`)

**Своя карточка ≠ публичный эндпоинт.** Вместо стыковки двух разных типов (`ProfileResponse` vs
`ContractorPublicResponse`) для редактирования «О себе» — своя карточка грузится ЦЕЛИКОМ через
`getProfile()` (тот же тип, что уже умеет редактировать `PortfolioSection`, без правок компонента
и без мерджа). Чужая — `getContractorPublic()`, read-only.

**Бейдж верификации — ТОЛЬКО для `verified`.** Для `pending`/`rejected`/`not_submitted` не
рендерится ничего — единое «не верифицирован» на три разных по смыслу состояния вводило бы в
заблуждение, а посторонним детали чужого статуса не нужны (решение пользователя).

**Пустые состояния — везде явный текст, не дыра:** рейтинг `null` → бейдж не рисуется вовсе (тот
же принцип, что уже у `RatingBadge`), «О себе» пусто → «Исполнитель пока не заполнил…», отзывов
нет → «Отзывов пока нет.».

**Lint-находка:** синхронный `setState` в начале эффекта (сброс перед фетчем) поймал
`react-hooks/set-state-in-effect`. Исправлено переходом на `key={contractorId}` (обёртка
`ContractorCardPage` → внутренний `ContractorCard`) — штатная замена, заодно чинит потенциальный
показ данных предыдущей карточки при переходе между двумя чужими профилями клиентским роутером.

**Доработки по итогам браузерной проверки (три правки):**
1. Пункт «Профиль» перенесён из горизонтального меню в выпадашку `AppNav` (`Профиль → Настройки →
   Выйти`, только у исполнителя) — `AppNavUser` дополнен `id`. `activeLink` для `/contractors/`
   убран из горизонтального меню целиком (там пункта больше нет).
2. `PortfolioSection` получил режим просмотр/редактирование: пустой сохранённый текст → сразу
   форма (как было), есть текст → читаемый блок + «Редактировать», «Отмена» откатывает черновик
   без сохранения (видна только когда есть что отменять).
3. Теги в отзывах — проверено запросом: уже были у одного отзыва (`contractor_id=50`), `ReviewDisplay`
   рендерит их корректно, бага не было — ничего чинить не пришлось.

**Пункт 3 (дата регистрации + счётчик заявок) — отдельная разведка, затем реализация:**
`marketplace.services.get_completed_counts()` — тот же паттерн, что `reputation.services.
get_ratings_for_contractors`, но в ОБРАТНУЮ сторону: `accounts` зовёт `marketplace`. Единственное
реверсивное ребро зависимости во всём графе модулей — осознанное решение пользователя после
разбора фактов (в отличие от `reputation`, которая уже зависела от `marketplace` через прямой
импорт модели в `ReviewDetailCreateView`, у `accounts` такой зависимости не было вообще ни разу).

**Найденный по ходу гэп:** `ProfileSerializer` не отдавал `date_joined`/`completed_requests_count`
(добавлены только в `ContractorPublicSerializer`) — своя карточка показала бы пусто. Добавлено в
оба сериализатора тем же способом.

Seed-даты `date_joined` у 4 демо-аккаунтов разнесены на 2025–2026 через shell (`.update()`, не
фикстура — в git не попадает), чтобы «на платформе с …» не показывал у всех одну дату.

Дата — `Intl.DateTimeFormat("ru-RU", {month:"long", year:"numeric"})`, первая буква заглавная
вручную (тот же приём, что уже `RequestRow.formatDate`/`ResultThread.formatDateTime`). Счётчик 0
показывается явно, не прячется.

**Находка при закрытии блока:** этап 5 и все доработки после него оказались не закоммичены
несколько заходов подряд (пользователь ожидал, что уже всё в git). Разбито на два коммита по
смыслу задним числом — `8db27ef` (вход + карточка) и `831248a` (backend-агрегат + режим
редактирования «О себе»), без хирургической резки одного диффа по хронологии реплик (риск
получить нерабочий промежуточный коммит).

Тесты на конец блока: accounts 33/33, полный прогон **175/175**. Миграции — только этап 1 и
задача 8, остальные этапы без изменений схемы.

## Чем закончили / что осталось

**Блок «Профиль» закрыт полностью** — настройки, публичная карточка, верификация с причиной
отклонения и новым статусом `not_submitted`. За пределами блока: email-уведомление об отказе

БД на территории РК, уведомления уполномоченного органа — юридическая задача, не программерская,
но блокирует реальный запуск. Зафиксировано в `docs/progress.md`.

**Мерж в `master` — отдельным подтверждением пользователя**, не в рамках этой сессии.
