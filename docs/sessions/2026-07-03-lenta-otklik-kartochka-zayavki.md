# Сессия 2026-07-03 — Лента заявок, отклик исполнителя, карточка заявки с картой

## Решение сессии

Продолжение шага B (после `2026-07-03-lenta-shag-a-model-lokatsii.md`, где была
исправлена модель домена). За эту сессию: (1) сама лента `/ru/feed` на реальных
данных, (2) доработки ленты по итогам ручной проверки, (3) отклик исполнителя —
сначала модалка, но она не подошла по UX, поэтому (4) заменена полноценной
страницей `/ru/requests/[id]` с картой MapLibre. Восемь коммитов, backend/frontend
разделены по каждому шагу, пересоздание/рестарт контейнеров между ними.

## 1. Лента заявок `/ru/feed`

Перед вёрсткой обнаружены два пробела в шаге A: у списка заявок не было
пагинации, у справочника `geo` не было ни одного read-эндпоинта.

**Backend:** `RequestListCreateView` — `PageNumberPagination` (`page_size=20`),
явный `order_by("-created_at")` (иначе DRF предупреждал об
`UnorderedObjectListWarning`). Новый `GET /api/geo/locations/` — справочник КАТО
одним деревом (`republican_cities` + `regions[{cities, districts}]`), фронт
запрашивает раз и строит каскад локально, без запроса на каждый уровень.

**Frontend:** `lib/api/{marketplace,geo}.ts` (типы + `getFeed()`/`getLocations()`),
`RequestRow.tsx` (строка + нейтральный `WorkTypeBadge` — тип работ это категория,
не состояние), `FilterBar.tsx` (тип работ + каскадный фильтр локации: республиканский
город, либо область → город/район), `feed/page.tsx` (сама лента: guard по роли —
contractor only, customer → `/dashboard`; deep-linking фильтров через query-параметры;
skeleton/два варианта пустоты/ошибка+повтор/401→login).

Попутно: два нарушения нового ESLint-правила `react-hooks/set-state-in-effect`
(default config Next 16) — заменены на паттерн react.dev «adjust state during
render» вместо синхронного `setState` в эффекте.

## 2. Доработки ленты по итогам ручной проверки

Пять точечных правок после того, как пользователь открыл `/ru/feed` в браузере:

- Глобальная стилизация скроллбаров (`globals.css`) — браузер рисовал нативную
  тёмную полосу по OS `color-scheme`, не в тон светлой темы.
- Колонка «№» в таблице ленты.
- Шапка ленты — счётчики «Заявок доступно» / «Новых сегодня». Потребовалась
  backend-правка: `RequestPagination.get_paginated_response()` считает
  `today_count` по тому же отфильтрованному queryset + `created_at__date=сегодня`.
- Текст подзаголовка ленты.
- Редирект после логина — по роли (`contractor→/feed`, `customer→/dashboard`),
  было всегда `/`.
- Отдельным заходом: счётчики переоформлены под референс-скриншот пользователя
  (синее акцентное число + двухстрочная подпись рядом, одна карточка с
  разделителем — вместо двух карточек с числом сверху/подписью снизу).

Зафиксированы (не реализованы) два открытых продуктовых вопроса — решать при
постройке кабинета заказчика: переименование `/dashboard` («Кабинет»/«Мои
заявки») и видимость общей ленты заказчику (соцдоказательство vs
конфиденциальность других заказчиков).

## 3. Отклик исполнителя — backend-подготовка

Перед формой отклика — план (Bid: только contractor, только open-заявка,
дубликат блокируется — проверено кодом и живым curl) выявил найденный curl'ом
баг: `raise ValidationError("Вы уже откликнулись...")` сериализуется DRF в голый
JSON-массив `["..."]` без ключа `detail` — фронтенд не находил сообщение.
Исправлено на `ValidationError({"detail": "..."})`.

Добавлен `has_bid` в `RequestFeedSerializer` — аннотация `Exists(Bid.objects.filter(
request=OuterRef("pk"), contractor=user))` в `get_queryset()` (один SQL-запрос на
всю ленту, не второй эндпоинт и не N+1) — чтобы кнопка в ленте сразу показывала
«Вы откликнулись» на заявках, где уже есть отклик.

## 4. Отклик через модалку — построена и заменена в этой же сессии

Сначала реализована модалка на нативном `<dialog>` (`Modal.tsx` + `BidModal.tsx`,
цена/срок/комментарий, инфо-алерт неверифицированному исполнителю — формулировка
мотивирующая, не блокирует отправку). После ручной проверки в браузере пользователь
решил, что модалка не подходит по UX — нужна полноценная страница заявки. Оба файла
удалены в следующем шаге (подтверждено `grep`: других импортов не было).

## 5. Карточка заявки `/ru/requests/[id]` + карта MapLibre

**Backend:** `RequestFeedDetailSerializer(RequestFeedSerializer)` — только для
`RequestDetailView`, не для ленты (список не раздуваем геометрией, которая там не
используется). Добавляет `site_geometry` (`GeometryField(source="site.geometry")`)
— **голая** GeoJSON-геометрия (`{"type":"Point","coordinates":[...]}`), НЕ
`Feature`: в отличие от `sites.SiteSerializer` (`GeoFeatureModelSerializer`), тут
нет обёртки `properties`. `sites`-app и его permissions не тронуты (`SiteDetailView`
остаётся `[IsCustomer]`) — геометрию исполнителю отдаёт `marketplace`, который уже
верно резолвит доступ (open-заявки + назначенные ему). `tz_file` уже отдавался как
рабочая presigned-ссылка MinIO (`PublicURLS3Storage`) — backend-правка не
понадобилась.

**Frontend:** первая карта во фронте — `maplibre-gl@^5.24.0`, растровые тайлы OSM
(TODO в коде: usage policy не разрешает продакшн-трафик, переключить на
self-hosted/MapTiler перед выходом на пользователей). `SiteMap.tsx` (generic,
принимает голую геометрию) — автовыбор слоя (точка → circle, полигон/линия →
fill+line), `fitBounds` по координатам, устойчив к двойному монтажу StrictMode
(флаг `cancelled` — `'load'` не трогает уже удалённую `map.remove()`'ом карту).
`BidForm.tsx` — та же форма/валидация из `BidModal.tsx`, без `<dialog>`-обвязки,
карточка в сайдбаре. `RequestRow.tsx` — вместо колбэка модалки строка ленты
кликабельна целиком, кнопка/бейдж — настоящий `Link` на `/requests/{id}`.

Страница: `id` через `useParams()` (не `useSearchParams()` — не требует
`<Suspense>`). Guard по роли на уровне страницы (contractor only, customer — в т.ч.
на своей заявке — редиректится на `/dashboard`, полный обзор заявки заказчиком не в
этой сессии). Двухколоночный layout тем же flex-приёмом, что `FilterBar.tsx`
(`flexWrap: wrap` + `flex: "N 1 XXXpx"`, без media-query). Состояния: свой skeleton
(не ленточный), «Заявка не найдена» (нативный 404 от `RequestDetailView`),
ошибка+повтор (derived-loading через `requestKey`), 401→`/login`.
