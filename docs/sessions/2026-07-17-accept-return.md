## С чего началось

Последний открытый пункт цикла «двусторонние вехи»: `accept`/`return` у заказчика при
`status=result_submitted` — backend (`AcceptView`/`ReturnView`) существовал с 1.5, фронта не было
вообще. Обязательное условие, согласованное ещё 2026-07-14: `return_note` — причина возврата,
без неё «вернуть на доработку» бессмысленно (исполнитель сдаст то же самое повторно).

## Разведка (backend), до кода

Прочитаны `AcceptView`/`ReturnView` целиком: оба уже используют `IsCustomer` + `.filter(...).update()`
(правило проекта без исключений). `Request.return_note` на модели — **не существовало** (подтверждено
чтением модели целиком, не спутано с `result_note`). `RequestSerializer`/`RequestFeedDetailSerializer`
уже отдавали `result_files`/`result_note` заказчику и победителю — фронт их просто нигде не рендерил.

## План (backend)

1. `return_note = models.TextField(blank=True)` — **перезаписывается** при каждом возврате, не
   накапливается. Прецедент — `result_note` уже ведёт себя так же (`SubmitResultView`). История
   возвратов, если понадобится — события/AuditLog, не поле.
2. `ReturnView` — `return_note` обязательна, 400 без неё, проверка СТРОГО после владения/статуса
   (404 раньше 400 — тот же порядок, что уже в `ConsiderBidView`).
3. `ResultReturned` получает `return_note` в payload. **Проверено грепом ДО правки dataclass** (по
   явному требованию — не «проверю по ходу»): `subscribe()` нигде в backend не вызывается, `publish()`
   сейчас no-op на любое событие в проекте. Добавление обязательного поля безопасно.
4. Раскрытие — та же пара условий, что уже у `result_files`/`result_note` (`RequestSerializer` для
   заказчика, `RequestFeedDetailSerializer` ветка `assigned_contractor_id == viewer.id` для победителя).

## Реализация и тесты

Миграция `0007_request_return_note`, `ReturnView` переписан, `ResultReturned.return_note`,
сериализаторы. 7 новых тестов: пустая причина/только пробелы → 400 (оба случая), `return` на
`accepted` → 404, `accept` на `awarded` → 404 (оба «перехода-мимо-статуса», явно запрошены), инвариант
№9 на `return_note` через `assertNotIn` (не только «отсутствует», а именно проверка отсутствия ключа),
полный повторный цикл `awarded→submit→result_submitted→return→awarded→submit→result_submitted` с
проверкой перезаписи причины (не конкатенации) и накопления файлов (2, не замена). Backend 90/90,
`check`/`makemigrations --check` чисты.

## Curl-baseline на dev-БД — до фронта, по требованию

Полный цикл под реальными `cust-inv9`/`winner-inv9`/`loser-inv9` (сброшены пароли на время сессии),
заявка **#38**: award → submit → `GET` под заказчиком (файлы+note видны, `return_note=""`) → `return`
без причины (400 с текстом) → `return` с причиной (200, `awarded`) → `GET` под победителем
(`return_note` видна дословно) → **`GET` под проигравшим — `return_note` отсутствует в ответе вообще**
(инвариант №9 на живых, не только тестовых данных) → повторная сдача (файлы накопились, 2) → `accept`
→ `accepted`.

**Найдена и исправлена находка (не в исходном скоупе): миграция `0007` создана, но не применена к
dev-БД.** `manage.py test` был зелёным (тестовая БД пересоздаётся из миграций на каждый прогон), а
`POST /requests/` на реальной dev-БД упал 500 (`column "return_note" does not exist`) — забыт `migrate`
после `makemigrations`. Исправлено (`migrate marketplace` применён). **Правило зафиксировано в
`CLAUDE.md`**: после `makemigrations` — `migrate` на dev-БД в контейнере ОБЯЗАТЕЛЬНО, в тот же заход,
иначе тихая дыра (юнит-тесты не ловят, первый живой запрос падает 500).

## Живая проверка (браузер) — вскрыла крупную продуктовую находку

Заявка **#39** (`cust-inv9`) доведена вручную через форму до `result_submitted`, затем пользователь сам
прошёл поток «Вернуть на доработку» в браузере до конца. **Кнопки технически работают корректно** —
заявка ушла в `awarded`, `return_note='Доработать'` сохранилась (подтверждено инвентаризацией БД после).

Но: карточка результата у заказчика **исчезает** после возврата (её условие рендера —
`status ∈ {result_submitted, accepted}`, `awarded` в этот список не входит). Пользователь хочет вместо
этого **ленту переписки** (сдал+коммент → вернул+коммент → исправил+коммент, с датами, ничего не
затирается) — текущая модель (`result_note`/`return_note` как одиночные перезаписываемые поля) для
этого структурно не годится.

**Инвентаризация данных заказчика (по запросу, факт, не предположение)** — 6 заявок у `cust-inv9`
(id 34–39), статусы/файлы/заметки показаны построчно в чате. Отдельно подтверждено: **заявки id=45,
на которую ссылался пользователь, не существует нигде в системе** (максимальный id в таблице — 39,
последняя заведённая в этой сессии).

**Подготовлено сравнение MIN/FULL** (не реализовано, кода не писал — решение за пользователем):
- **MIN** — одна небольшая модель `ResultEvent` (request/kind[submitted|returned]/note/created_at),
  автор вычисляется из `kind` (не отдельное поле — в системе ровно два участника на заявку). `ResultFile`
  получает nullable FK на `ResultEvent` — заодно закрывает старый техдолг «непонятно, какие файлы
  свежие после нескольких сдач». Оценка объёма — сопоставимо с Блоком 5 (новая модель, 2 переписанные
  вьюхи, сериализаторы, тесты, один новый фронт-компонент вместо двух старых).
- **FULL** — то же плюс явный `author` FK, свободные комментарии вне submit/return (реальный чат — не
  в `PRODUCT_SPEC`), завязка на ещё не построенный блок уведомлений (Волна 2). Риск спроектировать не
  ту форму до появления реального блока уведомлений.
- Инвариант №9 — не в зоне риска ни там, ни там: тот же уже дважды проверенный гейт
  `assigned_contractor_id == viewer.id`.
- Существующие `result_note`/`return_note` — историю по ним всё равно не восстановить (уже
  перезаписывались до этого решения) — предложено оставить как unused/deprecated, не спешить с
  миграцией удаления в ту же неделю, когда завели.

Три мелкие находки той же живой проверки, план готов (не реализовано, не зависят от выбора MIN/FULL):
1. Текст подтверждения приёмки — убрать «вернуть заявку на доработку станет нельзя».
2. `STATUS_LABELS`: `accepted` → «Закрыта», `result_submitted` → «Результат сдан, примите работу».
   Проверена связка с `Badge.tsx` (цвета по точным строкам лейбла, `STATUS_VARS`) — менять текст
   значит трогать оба файла синхронно; единственный потребитель — `/requests/my/page.tsx`.
3. Баннер «закрыта» у исполнителя (`MyBidStatusPanel`) — сейчас серый (`tone: "done"`, та же ветка,
   что у rejected — в этом и баг), фикс — `tone: "active"`, токен уже есть, совпадает с тем, что
   показывает заказчику `ResultReviewCard`.
