# Сессия 2026-07-03 — Лента заявок, шаг A: модель домена (цена/срок, локация, роли)

## Решение сессии

Планировали экран ленты заявок (`/ru/feed`) и на этапе плана вскрылось, что модель домена
не готова: цена и срок жили не там (их вообще не было), локация была плоским текстом, а
исполнителю и заказчику отдавались одни и те же поля заявки без разбора по роли. Решили
не строить экран поверх неправильной модели — сначала выправили backend (шаг A), сама лента
(шаг B) — следующая сессия.

## 1. Цена и срок — поля отклика, не заявки

Заказчик публикует только объём работ; цену и срок предлагает исполнитель при отклике —
заказчик выбирает из предложений. `Bid.price` (`DecimalField`), `Bid.deadline_days`
(`PositiveIntegerField`, срок в календарных днях, не дата — дата старта неизвестна до
`award`). Nullable на уровне модели (тесты и админка создают `Bid` напрямую), но обязательны
в `BidSerializer` при создании через API.

## 2. Локация — справочник КАТО вместо текста

`Request.city` (`CharField`) заменён на `location_type` (город/район) + FK на новые модели
`geo.City`/`geo.District`. Схема справочника (`apps/geo/models.py`):

```
Region (область, 17 шт.)
  ├── District (район, FK → Region)
  └── City     (город ОБЛАСТНОГО значения, FK → Region, nullable — null для городов
                РЕСПУБЛИКАНСКОГО значения: Астана/Алматы/Шымкент)
```

Область на `Request` не хранится напрямую — тянется через `city.region` или
`district.region`, чтобы не было рассинхрона. Справочник живёт в `apps/geo`, не в
`marketplace` — общий географический ресурс, пригодится и `sites` (адрес объекта).

**Наполнение** — data-миграция (`geo/migrations/0002_load_kato_data.py`), читающая
`geo/fixtures/kato_regions.json`, выполняется автоматически при `migrate` на любом
окружении (не ручная `loaddata`). Итог: **17 областей, 170 районов, 42 города** (39
областного значения + 3 республиканского). Источники — статьи ru.wikipedia.org
«Административно-территориальное деление N области» и официальная публикация stat.gov.kz
на 01.01.2025, **не память LLM** (состав областей менялся в 2022 — Абайская/Жетысуская/
Улытауская выделены из ВКО/Алматинской/Карагандинской). Города — строго официальный
статус «областного значения»: там, где официально один такой город (например
Петропавловск в СКО), в справочник вошёл только он, крупные города районного значения
намеренно не включены.

Данные собирались в два прохода: сначала вручную сверил 2 области (Акмолинская,
Северо-Казахстанская) через прямой WebSearch — и сразу поймал ошибку в примере
пользователя («Тайыншинский район» — это Северо-Казахстанская область, не Акмолинская,
как было предположено в постановке задачи). Остальные 15 областей собрал фоновый агент
(`general-purpose`, WebSearch) по тому же методу и источникам. Агент явно пометил 4 пункта,
где не смог однозначно подтвердить статус — не включены молча, а вынесены отдельно:

1. **Алатау** (Алматинская обл.) — город образован указом в январе 2024, статус особый
   (территория ускоренного развития), очень свежий.
2. **Байконур** (Кызылординская обл.) — формально город областного значения, но
   параллельно имеет статус по договору аренды с РФ.
3. **Улькен Нарынский и Маркакольский районы** (ВКО) — новые районы 2022–2023, итоговое
   число районов (11) подтверждено stat.gov.kz, материнские районы отдельно не сверены.
4. **Жанасемейский и Маканчинский районы** (Абайская обл.) — новые с 01.01.2024,
   подтверждены новостными источниками, не сверены со stat.gov.kz.

Все четыре сейчас в справочнике; пользователь сверит сам при случае — не блокер, поскольку
правки вносятся через Django Admin (`Region`/`District`/`City`) одной строкой, без новой
миграции.

Условная валидация локации (`city` обязателен при `location_type=city`, `district` — при
`district`, взаимоисключающе) — тем же приёмом, что ИИН/БИН в `accounts` (условно
обязательные поля по типу).

Побочный эффект миграции: 2 старые dev-заявки (id 1, 2, тестовые аккаунты `zakaz7@`/
`zakaz2@`) потеряли текстовый `city` — надёжного автосопоставления строка→справочник не
было. По решению пользователя удалены как мусор от старой модели.

## 3. Видимость полей заявки — по роли

Один эндпоинт (`GET /api/marketplace/requests/`), два сериализатора вместо условного
скрытия поля через `SerializerMethodField` — так на уровне типов невозможно случайно
утечь данные, если кто-то забудет проверить контекст:

- **`RequestSerializer`** (заказчик, свои заявки) — с `bids_count`, без `customer`.
- **`RequestFeedSerializer`** (исполнитель, открытая лента) — с `customer` (через
  `CustomerBriefSerializer`: открытая информация, не секрет, кто разместил заявку), БЕЗ
  `bids_count` — если исполнители видят «0 откликов», они понимают отсутствие конкуренции
  и завышают цену; скрытие защищает от манипуляции ценами. `status` тоже не отдаётся —
  лента и так только `open` (как только заявка `awarded`, она пропадает из ленты тем же
  queryset-фильтром, который уже был в коде до этой сессии).

`get_serializer_class()` в `RequestListCreateView`/`RequestDetailView`, ветвление по
`request.user.role`. По пути поймал и починил регрессию: `drf-spectacular` интроспектирует
схему с `AnonymousUser` (нет атрибута `role`) — заменил `self.request.user.role` на
`getattr(self.request.user, "role", None)`.

Фильтр в ленте — точные ID из справочника (`region_id`/`district_id`/`city_id`), не
текстовый `icontains`, раз данные теперь структурированы.

## Проверено

- **27/27 тестов** (было 19 — 8 новых: обязательность цены/срока, скрытие `bids_count` от
  исполнителя vs показ заказчику, условная валидация локации в обоих направлениях,
  исчезновение `awarded`-заявки из ленты).
- **Живой сквозной прогон через curl** (не только тесты — реальный dev-сервер,
  `localhost:8000`, с обходом проблемы кодировки кириллицы в этой Windows-shell через
  `--data-binary @файл` вместо инлайн `-d`): регистрация заказчика/исполнителя → заявка
  `location_type=city` → `location_display: "Алматы"`; заявка `location_type=district` →
  `location_display: "Акмолинская область, Аршалынский район"`; отклик с
  `price=250000.00, deadline_days=10` сохранился и вернулся в ответе; лента исполнителя —
  есть `customer`, нет `bids_count`, нет `status`; список заказчика — `bids_count: 1` есть,
  `customer` нет.
- `manage.py makemigrations --check` — миграции написаны руками, но совпадают с тем, что
  сгенерировал бы Django.
- `manage.py spectacular` — schema генерируется чисто (после фикса `AnonymousUser`);
  оставшиеся warnings (`AwardView`/`AcceptView`/`ReturnView`/`SubmitResultView`,
  `LogoutView`) — не моя правка, pre-existing, не трогал.
- Контейнер `backend` пришлось пересоздать один раз — `git stash`/`git stash pop` во время
  диагностики поймал autoreloader на промежуточном состоянии файлов, процесс завис на
  крашнутом импорте; `docker compose restart backend` починил.
