**Лейблы статуса для исполнителя-победителя — отдельный набор от `STATUS_LABELS`.**

`result_submitted` → «Сдано, ожидает приемки» (заказчицкое «Результат сдан, примите работу»
написано с позиции принимающего, не подходит сдающей стороне), `accepted` → «Закрыта» (не

конец цикла, слово должно совпадать с тем, что видит заказчик). `Badge.tsx`
(`STATUS_VARS`/`RequestStatus`) расширен одним новым лейблом-строкой «Сдано, ожидает приемки» с
теми же токенами цвета, что у заказчицкой пары («В работе»/«Закрыта» уже существовали — переиспользованы
как есть).

## Фильтры

По явной границе, установленной пользователем: **на `/requests/my-bids` — только локация**
(фильтр по статусу отклика НЕ делается — три метки `BidOutcomeBadge` не дают полезного разреза, а
разводить `selected`/`rejected` в фильтре раскрыло бы исход прямо в таблице, отменяя решение,
сознательно принятое в самом компоненте). **На `/requests/my-work` — локация + статус заявки**
(три реально достижимых значения — `open`/`under_review` структурно недостижимы для
`Bid.status=SELECTED`, в опциях фильтра не показаны). Тип работ — сознательно не добавлен
(специализация исполнителя обычно однородна, разрез бесполезен). Сортировка в UI не выведена
(данных мало, весь список виден на экране).

Backend: `city_id`/`district_id` добавлены в `BidRequestBriefSerializer` — до этого сериализатор
отдавал только строку `location_display`, каскадный `<Select>` по образцу `/feed` без структурных
id было не собрать. Безопасность проверена ФАКТАМИ до правки: сериализатор рендерится ровно в двух
местах (`BidOwnerSerializer.request` → `MyBidListView`; `BidRequestWithStatusSerializer` →
`MyAwardedBidSerializer` → `MyAwardedListView`), оба — `IsContractor` + `filter(contractor=
request.user, ...)`; `BidCustomerSerializer` (заказчик) поле `request` исключает из `Meta.fields`
целиком — до заказчика эти id никогда не доходят. `assertNumQueries`-тесты на N+1
(`test_my_bids_location_display_does_not_n_plus_one`,
`test_my_awarded_location_display_does_not_n_plus_one`) прогнаны после правки — зелёные, новых
запросов нет (ожидаемо: `city_id`/`district_id` — это FK-колонки самой модели `Request`, Django
подгружает их с базовой строкой без `select_related`).

Фильтрация — клиентская (`useMemo` над уже загруженным массивом), без URL query-параметров: у
самого нагруженного тестового исполнителя (`winner-inv9`, id 50) — 12 строк в `/my-bids/`, 10 в
`/my-awarded/` (проверено запросом к dev-БД, не по памяти). `EmptyState` — два варианта на каждом
экране, не один: `"no-data"` (данных нет вообще) и `"no-results"` (данные есть, фильтр отсёк всё),
тот же принцип, что на `/feed`. Кнопка «Сбросить фильтры» — только при активном фильтре.

**Найденный и исправленный в процессе баг единообразия:** фильтр локации на `my-bids` изначально
был написан тернарником (`cityId != null ? ... : ...`), на `my-work` — цепочкой `if`. Приведено к
одному стилю (`if`-цепочка) по прямому требованию — два соседних экрана не должны решать одну
задачу по-разному, даже при совпадающем поведении.

## Баг сброса `LocationCascadeSelect` (найден и обойдён, компонент не тронут)

Разведка кодом (без запуска), по прямому требованию — «покажи фактами, не рассуждением»:
`resolved.regionId` (`LocationCascadeSelect.tsx:58-74`) вычисляется только из внешнего
`cityId`/`districtId`; внутренний `pendingRegionId` (:80-85) синхронизируется с ним лишь когда
`resolved.regionId` реально меняется. При выборе области без завершения каскада (`handleLevel1Change`,
:107-110) наружу уходит `{null, null}` — то же самое значение, что было до выбора области. Клик
«Сбросить фильтры» вызывает `onChange({null, null})` повторно — для компонента это не изменение
props, синхронизация не срабатывает, `pendingRegionId` остаётся на старой области, второй `<Select>`
продолжает её показывать. Подтверждено построчным разбором кода, не запуском в браузере.

Фикс — **не в самом компоненте** (второй потребитель, `RequestForm.tsx`, не проверялся, шире
область поражения): форсированный `key`-remount на обеих страницах (`locationResetNonce`,
инкремент в `handleReset()`, `key={locationResetNonce}` на `<LocationCascadeSelect>`). Замечено и
устранено попутно несовпадение имени пропа между двумя страницами (`resetNonce` в `my-bids` vs
`locationResetNonce` в `my-work`) — приведено к одному имени.
