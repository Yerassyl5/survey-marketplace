## Блок 8 — тред переписки (`ResultEntry`) вместо поля

**Решение: FULL, но в аккуратном объёме.** Раз всё равно заводим новую сущность и переписываем
Submit/Return — разница между «сообщения+файлы по дате» и «файлы привязаны к сдаче» — одно FK-поле.
Взамен уходит хрупкая группировка файлов по времени задним числом.

**Что переигрывается из `0007`:** та миграция добавила `Request.return_note` — новая миграция `0008`
её не редактирует (уже запушена), а функционально замораживает: `Submit/Return/AcceptView`
перестают писать в `result_note`/`return_note`, поля остаются в схеме unused (удаление — отдельным
подшагом позже).

### Модель

`ResultEntry`: `request` FK, `author` FK, `kind` (`submitted`/`returned`/`accepted`), `text`,
`created_at`, `Meta.ordering=["created_at"]`. **Роль не хранится и не выводится сравнением
`author == assigned_contractor`** — `kind` уже однозначно определяет автора структурно (`submitted`
физически может создать только `SubmitResultView` под `IsContractor`, `returned`/`accepted` —
только `ReturnView`/`AcceptView` под `IsCustomer`); сравнение с текущим `assigned_contractor` было
бы к тому же неверным для `returned`/`accepted` (нужно обратное сравнение с `customer`) и уязвимо к
будущему переназначению исполнителя. `author` хранится явно — не ради роли, ради честной привязки
личности. `ResultFile` получил FK `event` на `ResultEntry` — файлы каждой сдачи привязаны к своей
записи, не к заявке целиком.

### Находка при реализации: `AcceptView` тоже должен создавать запись

Собственный тестовый сценарий («сдал→вернул→сдал→принял = 4 события») требовал, чтобы `AcceptView`
тоже писал `ResultEntry(kind=accepted, text="")` — иначе цикл давал бы только 3 события. Добавлено.

### Досдача без возврата — upsert, не второй пузырь

Реальный сценарий: сдал, забыл файл, досдаёт ДО реакции заказчика — это НЕ ошибка, блокировать
вредно. Но два submit-события подряд без возврата между ними в ленте читались бы странно.
**Решение:** `SubmitResultView` принимает `status ∈ {AWARDED, RESULT_SUBMITTED}`; при
`RESULT_SUBMITTED` файлы падают в последнее открытое `submitted`-событие, текст дописывается через
`\n\n` (не заменяется, не игнорируется — оба варианта тихо теряли бы часть текста), `created_at`
события НЕ обновляется (дата «начала» эпизода, у файлов свой `uploaded_at`). Возврат — единственное,
что рвёт серию на новое событие. **Гонка параллельных submit-запросов закрыта `select_for_update()`**
— сознательно строже, чем аналогичная гонка в `WithdrawBidView` (там остался MVP-компромисс без
блокировки): цена здесь выше — не лишний откат статуса, а реальное раздвоение истории сделки.

### Легаси-данные — вариант C, не бэкфилл

Три варианта обсуждались: (A) один синтетический `submitted` на все старые файлы — отклонён по
факту: заявка #38 (создана в Блоке 7 через curl) реально прошла `submit→return→submit`, 2 файла
разнесены на 51 секунду — синтез склеил бы их неверно, потеряв факт возврата между ними. (B)
группировка по `uploaded_at` — сложнее и хрупче, чем оправдано для нескольких строк dev-данных.
**(C) выбрано:** не бэкфиллить, пересоздавать тестовые сценарии через реальные эндпоинты (продакшена
нет). На конец сессии под тестовыми учётками осталось 6 старых заявок с файлами без `ResultEntry`
(id 34/35/37/38/39 + один сторонний `seed-customer`) — не пересоздавались (архивные, не было причины
трогать), их страницы честно покажут пустую ленту при наличии файлов — ожидаемое следствие решения
C, задокументировано с точными id, чтобы не спутать с багом.

### Инвариант №9 на треде — три независимых подтверждения

Юнит-тест `test_winner_sees_result_entries_loser_does_not` (именованный отдельно от
`test_return_note_visible_to_winner_not_to_loser`, по явному требованию — не полагаться на побочные
ассерты чужого теста), `assertNotIn` там же, и curl на живых данных (заявка #42, два разных

### Превью до кода — артефакты

Дважды строился превью-артефакт (тот же URL, republish): сначала лента 3 события + порядок секций
(эскиз), затем — по требованию — полная страница в новом порядке секций, 6 состояний ленты по
роли/моменту цикла (пусто/на проверке/после возврата/закрыто), явные решения по различению сторон
(без пузырей/аватаров — роль словом в заголовке + цветом kind-пилюли, второй параллельный код цвета
не заводили), по расположению действия (под лентой, как поле ввода под историей чата — причинность
читается хронологически, не «свежее сверху»), по пустому состоянию (без рельса ленты «в никуда»).
**По итогам превью баннер причины возврата над формой убран** — лента уже показывает последнюю
запись «Заказчик вернул: <причина>» прямо над формой, баннер дублировал те же слова. По той же
логике убран отдельный баннер «Результат принят, сделка закрыта».

### Урок сессии: мокап ≠ код

Превью-артефакт показывал верный порядок секций (Отклики → Результат), утверждён пользователем — но
при реализации фронта была отредактирована только НАЧИНКА карточек, физическая позиция `BidsPanel` в
JSX не была перенесена. Он остался отдельным сиблингом ПОСЛЕ двухколоночного ряда — визуально ниже
«Результата» (наоборот задуманному) и вне `flex:"2 1 480px"`-контейнера левой колонки (визуально
шире прочих карточек). Расхождение поймано только живой браузерной проверкой пользователем — не
раньше. Исправлено одним переносом (`BidsPanel` внутрь левой колонки), заодно решило обе жалобы
разом (порядок и ширина — одна причина). Зафиксировано в техдолге `docs/progress.md`: после
утверждения мокапа — явно переносить в реальный код и проверять именно в приложении, согласование
мокапа не равно завершению задачи.

### Косметика статусов (отдельный коммит от треда, тот же день)

`accepted` → изумруд: `--ds-success` получил РЕАЛЬНО отдельный оттенок (`#065F46`/`#ECFDF5`, до
этого — точная копия hex `--ds-active`, что делало «закрыто» неотличимым от «в работе»/«выбран»).
`awarded` («В работе») → индиго: новый токен `--ds-progress`, заведён специально под статус заявки —
`--ds-active` остался за статусом ОТКЛИКА («Выбран»/«Рассмотрен»/«Поздравляем»), смешивать два разных
смысла на одном токене было бы неверно. Оба оттенка выбраны через превью-артефакт (варианты A/Б
бок о бок с уже занятыми соседями, обе темы), не «на глаз». Лейблы `STATUS_LABELS`: `accepted` →
«Закрыта», `result_submitted` → «Результат сдан, примите работу» (правка синхронно в `Badge.tsx` —
цвета там по точным строкам лейбла). Текст диалога приёмки — без фразы «вернуть заявку на доработку
станет нельзя».
