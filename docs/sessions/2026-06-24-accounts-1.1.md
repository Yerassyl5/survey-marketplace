# Сессия 2026-06-24 — модуль accounts (1.1)

## Что сделано
- Реализован пункт 1.1 PRODUCT_SPEC: регистрация заказчика и исполнителя.
  - Кастомная модель `User` (логин по email, роли `customer`/`contractor`, ручной ввод ФИО/ИИН/БИН/телефон).
  - `ContractorProfile`: `license_number`, `attestation_number`, `license_expiry`, `verification_status` (по умолчанию `pending`), `verification_method`.
  - Эндпоинты `POST /api/accounts/register/customer/` и `/register/contractor/` (DRF), валидация ИИН/БИН по типу лица.
  - Доменное событие `UserRegistered`.
- Подключён drf-spectacular: `/api/docs/` (Swagger UI), `/api/schema/`.
- `.http`-файл с примерами запросов: `backend/apps/accounts/requests.http`.
- Найден и исправлен баг: ответ на регистрацию исполнителя не показывал `license_number`/`attestation_number` (поля физически на `ContractorProfile`, не на `User`) — исправлено через `to_representation`.
- Docs-only задачи: задачи бэкапа Postgres (progress.md + architecture.md §9), пункт про условные поля ИИН/БИН на форме регистрации фронтенда (progress.md).
- `.gitignore`: добавлены `.claude/` и `Архитектура.docx`.
- Репозиторий подключён к GitHub: `https://github.com/Yerassyl5/survey-marketplace.git`, ветка `master` запушена и привязана как upstream.

## Важное решение
Локальная Postgres-БД пересоздана (volume `project_postgres_data` удалён) для перехода на `AUTH_USER_MODEL` — миграции были только локальные dev-данные (суперюзер), без потери реальных данных. Новый суперюзер: `admin@eospatial.kz`.

## Проверка
Все эндпоинты регистрации и Swagger проверены через curl (см. сообщения сессии) — `201` на успешную регистрацию, `400` с понятной ошибкой на нарушение валидации ИИН/БИН по типу лица.
