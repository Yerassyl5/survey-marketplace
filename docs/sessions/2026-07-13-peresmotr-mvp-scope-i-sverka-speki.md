## Принятые продуктовые/архитектурные решения

1. **Рассмотрение откликов заказчиком** — `Bid.considered_at`, раскрытие телефона исполнителя
   в момент нажатия «рассмотреть», выбор исполнителя возможен только из рассмотренных откликов
   (минимум один обязателен).
2. **Новый статус заявки `under_review`** — наступает автоматически при первом отклике (не
   отдельным действием заказчика). Жизненный цикл: `open → under_review → awarded →
   result_submitted → accepted`.
3. **Отзыв заявки заказчиком** — только пока `status=open` (нет ни одного отклика). После
   первого отклика заказчик отозвать не может; единственный рычаг — администратор удаляет
   заявку через Django Admin.
4. **Репутация перенесена из Волны 2 в MVP (1.10)** — отзыв опционален, только после закрытия
   сделки: звёзды (1–5) + положительные теги-галочки из справочника + необязательный текст.
5. **Почта и уведомления перенесены из Волны 2 в MVP (1.11)** — подтверждение почты,
   сброс пароля, «вас рассматривают», «вас выбрали», результат верификации (почта+кабинет),
   заказчику письмо только на первый отклик. **Гейт почты — мягкая блокировка:** вход разрешён
   сразу, но создание заявки/отклик заблокированы до подтверждения; существующим пользователям —
   data-миграция (грандфазеринг, всем `True`).
6. **Dev-почта — Mailpit** в `docker-compose.yml` (тот же SMTP-код, что и в проде).
7. **AI-генерация ТЗ, упрощённая версия — в MVP (1.12)**: опросник → шаблонный промпт →
   черновик + обязательный дисклеймер. НЕ RAG (полная версия с цитируемостью норм — Волна 4+).
8. **Верификация исполнителя закреплена как постоянно опциональный процесс** — не временная
   мера MVP до включения блокировки, а решение навсегда: неверифицированный откликается без
   ограничений, статус виден заказчику в каждом отклике.
9. **Кабинеты** — заказчик: «Мои заявки» = открытые + архив; исполнитель: два раздельных
   раздела «Мои отклики» / «В работе и выполненные».

## Находка при ревью спеки: утечка статуса заявки через `updated_at`

Автопереход `open → under_review` при первом отклике создавал тот же канал утечки, что
платформа уже закрывает скрытием `bids_count` от исполнителя (защита от манипуляции ценами).
Если переход делать через `instance.save()`, `Request.updated_at` (`auto_now=True`) молча
обновится, а фид-сериализатор отдаёт `updated_at` в сыром JSON — исполнитель мог бы через
голый curl/devtools понять «кто-то откликнулся», не видя явного статуса.

Решение, зафиксированное в спеке (новый инвариант №9):
- `FEED_VISIBLE_STATUSES = (open, under_review)` — единая константа во всех точках входа
  исполнителя, не раздельная фильтрация по местам кода.
- Переход `open → under_review` — только через queryset `.update()`, никогда `.save()`.
- `updated_at` отдельно исключён из фид-сериализаторов (двойная защита).
