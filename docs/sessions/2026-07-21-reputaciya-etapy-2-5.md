## Этап 2 — эндпоинты создания и чтения отзыва (коммит `fd19dea`)

Разведка (без кода) прошла по существующим паттернам `AcceptView`/`ReturnView`/`ConsiderBidView`
— гейт «владение + статус неразличимы» через один `filter().first()` → 404, отдельно от него
конфликт состояния (дубль) → 409, тот же принцип, что уже в `ConsiderBidView`/`WithdrawBidView`.

**Главный архитектурный вопрос сессии-предшественницы (этап 1→2):** отзыв — отдельный ресурс
(`GET /api/reputation/requests/{id}/review/`) или поле внутри ответа marketplace? Выбран **отдельный
эндпоинт** — не заводить новую зависимость `marketplace → reputation` (сейчас зависимость только в
одну сторону, `reputation → marketplace` через FK), и отзыв переиспользуется на будущей карточке
исполнителя (Волна 2), не только на странице заявки.

**Видимость GET — решение, опирающееся на инвариант №9 новой редакции:** `permission_classes =
[IsAuthenticated]`, БЕЗ проверки владения/роли — отзыв публичен по решению продукта («скрывать
нельзя», доверие строится на том, что отзывы реальны). 404 — только когда отзыва физически нет
(`Review` существует исключительно при `Request.status=accepted`).

**POST** — только заказчик-владелец на `accepted`-заявке, паттерн `AcceptView` (один
`filter(pk=.., customer=.., status=ACCEPTED).first()` → 404). Два доп. 409-кейса, найденные при
ревью: дубль отзыва (заказчик и так знает, что заявка есть и отзыв уже оставлен — 404 сбивал бы
фронт с толку) и **`assigned_contractor_id is None`** — `Request.assigned_contractor` это
`SET_NULL`, а `Review.contractor` не nullable; без явной проверки `serializer.save()` упал бы 500
вместо внятного ответа при удалённом аккаунте исполнителя. Оба случая покрыты тестами
(`test_duplicate_review_returns_409`, `test_create_rejected_when_contractor_unavailable` —
последний реально удаляет исполнителя в тесте и проверяет 409, не только код).

Факт, добытый тестом, а не предположением: `rating` вне диапазона 1–5 → `400`, не `500` —
`ModelSerializer` автоматически подтягивает `MinValueValidator`/`MaxValueValidator` с модели, без
дублирования валидации в сериализаторе.

Тесты: reputation 24/24 после этапа. Миграций не было (модель не менялась).

## Этап 3 — агрегат рейтинга в карточке отклика (коммит `f1eab16`)

Рейтинг нужен ВНУТРИ ответа marketplace (карточка отклика, `BidsPanel`) — здесь, в отличие от
этапа 2, разрыв на отдельный запрос не подходит (не получить агрегат на каждый из N откликов
отдельным вызовом без N+1). Разобрали architecture.md §1: «внутренняя граница модуль↔модуль =
публичные сервисы модулей в памяти» — прямой импорт `reputation.models.Review` в
`marketplace/views.py` нарушил бы границу, легитимный путь — публичная функция.

`reputation/services.py::get_ratings_for_contractors(ids) -> dict[int, RatingData]` —
`RatingData` заведён явным `@dataclass(frozen=True)` (не голый dict/кортеж) по прямому требованию:
«граница модуля должна быть описана типом». `marketplace/views.py` импортирует только функцию и
датакласс, не модель. Один агрегатный запрос на весь список id (`Avg`/`Count` с `GROUP BY
contractor_id`), не по одному на исполнителя — тот же принцип, что уже `today_count` в
`RequestPagination`. `BidListCreateView.list()` переопределён (без пагинации у этой вьюхи
`ListModelMixin.list()` и так делает то же самое), чтобы `get_queryset()` не вызывался дважды
(владение + Bid-запрос задвоились бы).

**`float()` явно** в `RatingData.avg` — без этого `Avg()` на PostgreSQL мог бы отдать `Decimal`, а
DRF `JSONEncoder` сериализует `Decimal` как строку, не число. Проверено тестом на сырых байтах
ответа (`r.content`, не `r.data` — на уровне `Response.data` Decimal/float уже неразличимы).

Формат в карточке — `★ avg (count)`, при нуле отзывов — поле `rating: null` (не отсутствие поля:
на фронте `X | null` заставляет обработать случай, `X?: T` легко проглядеть), бейдж не рисуется
вовсе. N+1 закрыт тестом по образцу `test_my_bids_location_display_does_not_n_plus_one`
(`CaptureQueriesContext`, 1 отклик vs 4 — число запросов не растёт).

Тесты: marketplace 91/91, reputation 24/24, полный прогон 126/126.

## Этап 4 — форма отзыва у заказчика + справочник тегов (коммит `610fc45`)

Backend: `GET /api/reputation/tags/` — простой `ListAPIView`, без пагинации (6 записей, тот же
принцип, что `GET /geo/locations/`), `IsAuthenticated`.

Frontend, разведка перед кодом: `ResultReviewCard.tsx` — образец паттерна (POST в `try/catch`,
`AuthRequiredError` → редирект, коллбэк без параметров, родитель обновляет данные). Но
`handleRequestChanged` в `page.tsx` — **полный рефетч** `GET /requests/{id}/`, который НЕ включает
отзыв (отдельный эндпоинт с этапа 2) — карточке отзыва понадобился собственный способ
переключиться в read-only. Star-рейтинга и чекбоксов в проекте нет вообще (grep по `components/ui`
— ни одного компонента), сделаны с нуля.

**`ReviewDisplay.tsx`** выделен отдельным презентационным компонентом (принимает только `review`,
ничего не знает про роль/владение) — специально заложено на этап 5, чтобы карточка исполнителя
переиспользовала его без единой правки.

**`ReviewCard.tsx`** — три состояния (`undefined`/`null`/`Review`), `GET` уходит из
`useEffect` компонента, но триггерится только когда `page.tsx` рендерит карточку
(`isOwnerView && status === "accepted"`) — компонента физически нет в дереве раньше, `useEffect`
не может выстрелить раньше. **Теги грузятся лениво** — только после того, как первый `GET`
подтвердил 404 (отзыва нет, форма реально нужна), не параллельно с ним. После `POST` — переключение
в read-only из ответа мутации напрямую, без повторного `GET`.

Звёзды — снять оценку нельзя (кнопка задизейблена, пока `rating` не выбран), теги — кликабельные
пилюли без лимита количества (в проекте нет чекбоксов, пилюли — устоявшийся визуальный язык).

**Побочный баг, найденный живой проверкой (не в исходном скоупе этапа):** длинный текст без
пробелов (комментарий на 2000 символов) не переносился, растягивал страницу горизонтально.
`overflowWrap: "anywhere"` отсутствовал на ВСЕХ шести местах в проекте, где рендерится свободный
пользовательский текст — добавлен везде одинаково: `ReviewDisplay.tsx`, `BidsPanel.tsx` (комментарий
отклика), `MyBidStatusPanel.tsx` (тот же комментарий, второе место рендера), `page.tsx` (описание
заявки И условия заказчика — второе тоже не было названо в исходном запросе, но того же класса),
`ResultThread.tsx` (лента сдачи/возврата результата).

Тесты: reputation 30/30, полный прогон 132/132.

## Этап 5 — отзыв у исполнителя-победителя (коммит `cfbbf6e`)

Разведка: условие «это победитель» на странице — `request.my_bid?.status === "selected"` (не
`isOwnerView`, та переменная только для заказчика). Факт, определивший главное решение этапа:
**проигравший исполнитель не получает от backend вообще ничего о судьбе заявки после `accepted`**
— `RequestFeedDetailSerializer.to_representation` раскрывает `status`/`result_files`/
`result_entries` только при `assigned_contractor_id == viewer.id`, значит для проигравшего
`request.status` структурно `undefined`, условие рендера карточки отзыва («если заявка accepted»)
буквально не на чем построить, а не только «не хочется показывать».

**`MyReviewCard.tsx`** — тонкая read-only обёртка: свой `GET`, но БЕЗ формы (создаёт только
заказчик), переиспользует `ReviewDisplay` без единой правки — ровно так, как заложено этапом 4.
Рендерится под `ResultSubmissionCard`, условие `my_bid.status === "selected" && status ===
"accepted"`.

**Правка по ходу (до реализации):** при `review === null` (отзыва нет) не рендерить НИЧЕГО — ни
карточки, ни текста-заглушки «пока не оставили». Причина в двух частях: отзыв полностью опционален
(PRODUCT_SPEC 1.10, инвариант №8) — «нет отзыва» может быть постоянным состоянием, не «ещё не
появился»; и тот же принцип, что `RatingBadge` на этапе 3 — отсутствие не должно выглядеть как факт.
`if (!review) return null` покрывает оба состояния (`undefined` и `null`) одной строкой.

Backend не менялся.
