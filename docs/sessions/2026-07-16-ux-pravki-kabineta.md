> Продолжение блока «кабинет исполнителя» (блоки 1–5 закрыты сессиями 2026-07-14/2026-07-15, см.
> `docs/sessions/2026-07-14-kabinet-ispolnitelya.md`, `docs/sessions/2026-07-15-sdacha-rezultata.md`).
> Ветка: `feature/consideration-foundation` (не смержена в `master`).

## С чего началось

Четыре косметических/UX-правки поверх уже готового цикла «сдача результата» (backend не менялся):
текст победителя, вид блока ТЗ на странице заявки, цвет метки «Подведены итоги», подсказка у
проигравшего про причину отказа. Три из четырёх — чистая косметика; тексты для правки 1 (текст
победителя) и правки 4 (подсказки при проигрыше) были согласованы пользователем дословно заранее.

## Что сделали

**Правка 1 — текст победителя (`MyBidStatusPanel`, ветка `selected && requestStatus === "awarded"`).**
Было: одна строка «Вас выбрали исполнителем». Стало: заголовок «Поздравляем, вас выбрали
исполнителем!» + абзац про сдачу результата (форма — `ResultSubmissionCard` ниже на той же странице).
`getStatusMessage` расширен с `{ text, tone }` до `{ heading, body?, tooltip?, tone }` — проверено, что
остальные ветки (rejected×2, pending×2, result_submitted/accepted) визуально не изменились (та же
однострочная разметка, просто `span` вместо голого текста в `div`).

**Правка 2 — блок ТЗ на странице заявки.** Было: кнопка «Скачать ТЗ», визуально терялась (похожа на
«файла нет»). Стало: строка с именем файла, как «Сданные файлы» у результата. Разведка фактом (dev-БД):
`tz_file` на модели — голый `FileField`, API отдаёт просто URL-строку, без `original_name` (в отличие
от `ResultFile`). MinIO `file_overwrite=True` (не задан `AWS_S3_FILE_OVERWRITE`, дефолт django-storages)
— значит basename URL почти всегда совпадает с очищенным оригинальным именем файла. Написана
`fileNameFromUrl()` (basename до query-параметров подписи, `decodeURIComponent`, `try/catch`) с
фолбэком на «Техническое задание». Проверено на реальных данных: заявка id=35 (кириллица,
«Счет_на_оплату__48_от_14_июля_2026_г.pdf») и id=16 (латиница, «4892731.pdf») — оба декодируются верно;
заявка id=32 (`tz_file` пуст на модели) — сериализатор отдаёт `tz_file: null`, ветка «Файл не приложен»
не сломана.
Общий презентационный атом `components/ui/FileLink.tsx` (иконка+имя+ссылка) вынесен из
`ResultFileList` (`ResultSubmissionCard.tsx`) и переиспользован в блоке ТЗ — разметка перенесена
1-в-1 (тот же `svg`, те же стили), визуальной регрессии в «Сданных файлах» нет (проверено в браузере
после переноса, раньше проверки ТЗ).

**Правка 3 — цвет метки «Подведены итоги» (`BidOutcomeBadge`, таблица `/requests/my-bids`).** Было:
тот же серый токен, что у «Ожидает рассмотрения» — визуально путались. Стало: `--ds-new` (синий, из
статусной палитры `globals.css`, ранее не занят в этом компоненте). Не зелёный (`--ds-active` = победа,
а итог мог быть проигрышем), не `--ds-review` (занят «На рассмотрении»), не raw `--ds-blue`/
`--ds-blue-light` (это CTA/ссылочные токены, другая смысловая семья). Метка сознательно не несёт исход
— таблица не должна намекать, выиграл исполнитель или нет.

**Правка 4 — `InfoTooltip` (новый, `components/ui/InfoTooltip.tsx`).** Иконка «?» у проигравшего в
`MyBidStatusPanel`, по hover/focus объясняет, ПОЧЕМУ не сложилось. Два текста по `considered_at`, не по
факту раскрытия цены/срока/комментария — это заказчик видит у ВСЕХ откликов всегда; `considered_at`
означает раскрытие ТЕЛЕФОНА (см. `ConsiderBidView`), то есть «заинтересовало достаточно, чтобы
связаться», а не «увидел цену». Тексты объясняют механику выбора, не утверждают «вы сделали ошибку» и
не советуют демпинговать (согласованы пользователем дословно, в код внесена только грамматическая
правка «цена и срок вас уже устроили заказчика» → «цена и срок уже устроили заказчика»).
a11y и позиционирование:
- триггер — `<button>` (в tab-order), не `div`/`span`;
- открытие по `mouseenter`/`focus`, закрытие по `mouseleave`/`blur`/`Escape`, плюс `onClick`-toggle как
  явный фолбэк для тача (фокус по тапу ненадёжен на iOS Safari);
- закрытая панель — `visibility:hidden` (не только `opacity`, тот сам по себе не убирает элемент из
  hit-testing и из дерева доступности) + `pointer-events:none` **постоянно** (панель без интерактивного
  контента, ловить мышь ей незачем в любом состоянии) + `aria-hidden={!open}` сверху — тройная защита
  от случайного залипания/чтения скринридером как всегда видимой;
- `role="tooltip"` на панели, `aria-describedby` на триггере, `aria-label="Почему так?"` (не общее
  «Пояснение» — избегает звучания дублирования с текстом подсказки);
- позиционирование — компонент НЕ создаёт свой `position:relative`-враппер под размер иконки: панель
  (`left:0; right:0`) растягивается на ширину ближайшего спозиционированного предка. В
  `MyBidStatusPanel` это строка «heading+иконка», которая сама растянута на всю ширину баннера
  (`flex-column`, `align-items: stretch` по умолчанию) — решает обрезание длинного текста (~300
  символов) в узком сайдбаре 320px без центрирования относительно иконки и без риска вылезти за
  вьюпорт сбоку.

## Важная находка: хостовый `tsc` недостоверен

После правки 1 `tsc --noEmit`, запущенный на хосте, показал ошибку `variant does not exist on
ButtonProps` на кнопке «Отозвать отклик» — не в изменённом коде. По требованию пользователя проверено
фактом, не предположением: `git stash` (застэшены все правки сессии) → `docker compose exec frontend
npx tsc --noEmit` **внутри контейнера** на чистом дереве — **пусто, ни одной ошибки**. `git stash pop`
→ тот же прогон с правками — тоже пусто. Хостовый `npx tsc` отдельно (без контейнера) показал 25+ ошибок
даже на чистом дереве — `Cannot find module 'maplibre-gl'`, `@radix-ui/react-slot`, `clsx` и другие:
хостовый `node_modules` неполный/не тот, резолвит не все зависимости фронтенда. Вывод: ошибка `variant`
была чистым артефактом хостового прогона, не фактом о коде; более того, недостоверность могла быть и в
части прошлых сессий, если `tsc` там гонялся так же на хосте.
Зафиксировано:
- `CLAUDE.md` → «Проверка результата»: `tsc --noEmit`/`npm run lint` фронтенда — ТОЛЬКО через
  `docker compose exec frontend`, никогда на хосте.
- `docs/progress.md` → «Техдолг фронтенда»: заметка про возможную недостоверность части прошлых
  прогонов + напоминание прогнать полный `tsc` в контейнере как бейзлайн при следующем крупном
  рефакторинге фронта.

## Проверка

`tsc --noEmit`/`npm run lint` — только внутри контейнера, чисто на каждом шаге (тот же preexisting
набор ошибок/warning в `AppNav.tsx`/`Screen.tsx`, не в изменённых файлах). Живая проверка в браузере
после каждой из четырёх правок отдельно (`docker compose restart frontend` между шагами), под тремя
учётками (`cust-inv9`/`winner-inv9`/`loser-inv9`), заявки #34 (`rejected`, `considered_at=None`) и #35
(`rejected`, `considered_at` есть — обе ветки `InfoTooltip` и кириллическое имя ТЗ проверены на живых
данных без создания новых через эндпоинты).
