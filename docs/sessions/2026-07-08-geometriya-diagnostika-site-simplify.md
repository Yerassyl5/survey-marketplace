# Сессия 2026-07-07 → 2026-07-08 — Диагностика геометрии, два бага с картой, упрощение Site, устранение точки-заглушки

## Решение сессии

Началось с жалобы «карта объекта иногда не грузится у заказчика» — цепочка из пяти
диагностических/фикс-циклов, каждый строго по схеме «сначала факты (read-only), потом план,
потом код+тесты+diff, потом коммит по явному разрешению»: (1) GeoJSON из QGIS в непроекции WGS84
ломал карту молча, (2) заказчик вообще не получал геометрию на своей же заявке, (3) упрощение
модели `Site` (убраны `address`/`cadastral_number`, объект больше не переиспользуется между
заявками), (4) найден и устранён скрытый дефект точки-заглушки при создании `Site` из файла,
(5) попутный React-warning от той же правки. 8 коммитов на `feature/site-simplify` (плюс более
ранние на `master`).

## 1. Диагностика: GeoJSON из QGIS ломал карту заказчика (коммит `0a7588e` на `master`)

Симптом: «Геометрия объекта не загружена», в консоли `Invalid LngLat latitude value`. Реальные
данные в БД показали: `Site id=16` («УВС2») хранил координаты вида `(703345, 5795889)` —
метры/UTM, но помечен `SRID=4326` (градусы). Причина — `geo/services.py::_parse_geojson`
парсила координаты вручную через голый `GEOSGeometry()`, слепо ставя `srid=4326` независимо от
реальной CRS файла; KML был безопасен по спецификации формата (всегда WGS84), GeoJSON — нет.

Фикс: GeoJSON теперь читается тем же GDAL `DataSource`-путём, что и KML — `layer.srs` берёт CRS
из `crs`-члена файла (реально проверено на установленном в контейнере GDAL 3.10.3 через
`ogr2ogr`: и длинная форма `urn:ogc:def:crs:EPSG::NNNNN`, и короткая `EPSG:NNNNN` распознаются
корректно) с явным `geom.transform(4326)`. Плюс bbox-валидация после `_force_2d`, общая для
любого источника — координаты вне физического диапазона WGS84 отклоняются с понятным текстом.
`SiteMap.tsx` получил try/catch вокруг `fitBounds` и отдельное состояние «Некорректные координаты
объекта» вместо ложного «не загружена». Backend 46/46, `tsc --noEmit` чисто.

## 2. Диагностика: заказчик не видел geometry на СВОЕЙ заявке (коммит `d1e4b41`)

Отдельный баг того же симптома, другая причина: `RequestSerializer` (единственный сериализатор
для СВОЕЙ заявки заказчика — и сразу после создания, и при просмотре) не включал поле
`site_geometry` в `Meta.fields` вовсе, хотя `RequestFeedDetailSerializer` (исполнитель) и
`RequestFeedForCustomerDetailSerializer` (заказчик, чужая заявка через `?scope=feed`) его уже
отдавали. Геометрия не доходила до фронта никак. Фикс — добавить то же поле
(`GeometryField(source="site.geometry")`) в `RequestSerializer`. Заодно TS-тип
`FeedRequestDetail`/`FeedRequest` помечен `optional` (не только `nullable`) — один эндпоинт
обслуживают три разных Django-сериализатора с разным набором полей, и эта самая правка типа сразу
поймала реальную точку потребления (`?? null` понадобился в `requests/[id]/page.tsx`).
Curl подтвердил: `site_geometry` теперь приходит владельцу с корректными координатами.

## 3. Упрощение `Site`: `address`/`cadastral_number` убраны (ветка `feature/site-simplify`, коммит `e823b32`)

Перед удалением — отдельная диагностическая сессия собрала полную карту ссылок на оба поля по
всему проекту (backend + frontend), с явной пометкой мест, которые сломались бы первыми
(`Site.__str__`, `SiteAdmin`, `SiteSerializer.Meta.fields`, `marketplace/tests.py::make_site()`,
вся секция «Объект» формы заявки). Реализация — одним заходом (модель + миграция
`0002_remove_site_address_remove_site_cadastral_number` + сериализатор + admin + тест + весь
фронтенд), чтобы не было промежуточного состояния «БД без колонок, код их ещё требует».

Заодно убран выбор СУЩЕСТВУЮЩЕГО `Site` — форма создания заявки теперь всегда создаёт новый
(участок определяется только геометрией). Секция «Объект» → «Участок». Попутно (коммит тот же
день): колонка ленты «Примечание» → «Описание» (показывала `contractor_note`, теперь
`description`; сами поля/API не тронуты).

Backend 46/46, `migrate` — чисто и идемпотентно, `manage.py check` без ошибок.

## 4. Найден и устранён дефект точки-заглушки (коммиты `9fc4e06`, `bf4b6c0`, `b2da7a9`)

При реализации упрощения (п.3) форма создания заявки, если заказчик прикладывал только файл (без
клика по карте), создавала `Site` сразу с фейковой точкой `[71.4, 51.1]` (центр Казахстана,
`FALLBACK_CENTER`) — потому что `Site.geometry` `NOT NULL` в БД, а серверный парсинг файла
(`uploadSiteGeometry`) требовал уже существующий объект и был вторым отдельным запросом. При
сбое второго запроса (или уходе пользователя со страницы) в БД мог остаться `Site` с
правдоподобной, но полностью фейковой геометрией — bbox-валидация её пропускает молча. Проверка
дев-БД на момент находки — 0 таких сирот, но риск был реальный и незаметный.

Разобрано и представлено пользователю на выбор 3 архитектурных варианта устранения:
(A) `geometry` становится nullable + честный `NULL` вместо фейка, (B) один атомарный
multipart-запрос в `SiteListCreateView` (файл разбирается внутри создания), (C) изолированный
бесстейтовый parse-эндпоинт, вызываемый ДО создания `Site`. Выбран **вариант C** — не трогает
`SiteListCreateView`/`SiteSerializer`/`SiteGeometryUploadView`/модель `Site` вовсе.

**Backend (`9fc4e06`):** `POST /api/geo/parse-geometry/` (`geo/views.py::GeometryParseView`) —
бесстейтовый, переиспользует существующий `parse_geo_file()` без изменений, возвращает
`{geometry, format}`, ничего не пишет в БД. Ошибки явно оборачиваются в `{"detail": "..."}`
(иначе DRF отдал бы голый JSON-массив — тот же анти-паттерн, что раньше чинили в
`BidListCreateView`). 4 новых теста, backend 50/50. Живой curl подтвердил репроекцию по
`crs`-члену и то, что `Site.objects.count()` не меняется до/после вызова (14 и 14).

**Frontend, UX-обсуждение перед кодом:** предложено 3 варианта совмещения интерактивного ввода
точки (`MapPointPicker`) и read-only показа геометрии из файла (`SiteMap`) в одной секции —
выбран вариант «условный рендер одного из двух готовых компонентов» как наименее рискованный для
уже рабочего пути «только точка» (оба компонента остаются нетронутыми).

**Frontend, реализация (`bf4b6c0`):** `SiteFields.tsx` сам вызывает `parseGeometryFile()` при
выборе файла (эффект на `value.file`, `valueRef`-паттерн против гонки состояний — та же техника,
что уже была в `MapPointPicker.tsx`) и получает финальную геометрию ДО `createSite()` —
`FALLBACK_CENTER`/`resolveInitialGeometry` удалены полностью. Карта переключается между
`MapPointPicker` (нет файла) / нейтральным лоадером «Проверяем файл…» (идёт парсинг) / `SiteMap`
с автозумом (файл распарсен) / обратно на `MapPointPicker` с сохранённой точкой (400 — файл не
стирает точку). `RequestForm.tsx`: второй сетевой шаг и завязанный на него ретрай-стейт
(`createdSiteId`, `siteGeometryUploaded`) убраны — `createSite`/`createRequest` вызываются по
одному разу. Сознательно принятый побочный эффект: без ретрая, если `createRequest` упадёт ПОСЛЕ
успешного `createSite`, повторный сабмит создаст второй (уже корректный) `Site` — не то же самое,
что чинили, но стоит иметь в виду отдельно.

**Побочный фикс (`b2da7a9`):** React-warning «does not recognize the hasError prop on a DOM
element» — `FormField` клонирует ребёнка с `id`/`aria-describedby`/`hasError`, а ветка-лоадер
«Проверяем файл…» была голым `<div>` (DOM-узел, не компонент, откуда и утечка пропа). Заведён
`MapLoadingPlaceholder` — принимает `id`/`aria-describedby` (a11y не потеряна), `hasError` не
объявляет («идёт парсинг» — не состояние ошибки). Визуальная подсветка ошибок в
`MapPointPicker`/`FilePicker` проверена отдельно чтением кода — работала и раньше, баг её не
касался.
