## О проекте (кратко)

Двусторонний маркетплейс инженерных изысканий для Казахстана. Соединяет **заказчиков** (проектировщики, девелоперы, застройщики, частные лица) с проверенными **исполнителями** (геодезисты, геологи, геофизики).

Механика: заказчик публикует заявку → исполнители откликаются → заказчик выбирает → работа выполняется → заказчик подтверждает результат → возможна оценка. Монетизация — подписка для исполнителей (включается позже по feature-toggle). Расчёты за услугу идут мимо платформы.

**Стиль:** модульный монолит на Django (НЕ микросервисы). Полная архитектура и список модулей — в `docs/architecture.md`. **Прочитай его перед архитектурными задачами.**

## Правила работы

### Сессии
- В начале каждой сессии читай `docs/progress.md` чтобы понять текущий статус проекта
- После выполнения любой задачи обновляй `docs/progress.md` — отмечай выполненные пункты [x], добавляй новые если появились
- Перед завершением сессии сохраняй краткое резюме в `docs/sessions/YYYY-MM-DD-название.md`
- Когда я напишу "заверши сессию" — обнови `docs/progress.md` и сохрани резюме сессии

### Планирование
- Перед любой реализацией объясни план: какие файлы создашь/изменишь, какой подход используешь
- Не пиши код пока я не одобрю план
- Одна задача = максимум 3-5 файлов за раз
- Одна задача = одна сессия, затем /clear
- Двигайся по пунктам `PRODUCT_SPEC.md` (например «сделай 1.3 по критериям приёмки»), а не целыми волнами за раз

### Git
- Делать коммит перед каждой крупной задачей
- Каждую новую фичу разрабатывать в отдельной ветке: `feature/название`
- Формат коммитов: на русском языке, кратко и по делу
- Перед входом в Auto-Accept режим всегда делать git commit

### Код
- Код и переменные: английский
- Комментарии: русский
- Коммиты: русский

## Инфраструктура
- Вся разработка ведётся в Docker локально
- docker-compose.yml должен работать одинаково локально и на сервере
- Секреты хранятся в `.env` (не в git)
- `.env.example` всегда актуален и закоммичен
- БД — PostgreSQL с расширениями **PostGIS** и **pgvector** (нужен кастомный Dockerfile, стандартный шаблон их не содержит). Django-образ требует системные GDAL/GEOS/PROJ.

## Безопасность
- `.env` всегда в `.gitignore`
- Никаких секретов, паролей, API ключей в коде или коммитах
- При создании проекта сразу создавать `.gitignore` с: `.env`, `__pycache__/`, `node_modules/`
- Валидация и санитизация всех загружаемых файлов (гео-файлы, документы, сканы лицензий)
- Файлы — в MinIO, в БД только ссылки. Персональные данные не уходят во внешние API.

## Деструктивные операции (обязательны, без исключений)

На этой машине помимо ПроГео есть **другие проекты** (Docker-тома, контейнеры, образы, файлы).
Инцидент 2026-07: глобальный `docker volume prune -f` безвозвратно удалил ~5GB данных
чужого проекта при попытке почистить один том ПроГео.

1. Никогда не выполнять команды с глобальным/массовым охватом, в том числе:
   `docker volume prune` / `docker system prune` / `docker image prune` / `docker container prune`
   (в любом виде, с любыми флагами); `docker volume rm` / `docker rm` / `docker rmi` по маске
   или списку; `rm -rf` по каталогу, содержащему больше чем целевой объект; `TRUNCATE`,
   `DROP DATABASE`, `DROP SCHEMA`, массовый `DELETE` без `WHERE`; `git clean -fdx`,
   `git reset --hard` без явной причины.
2. Удалять можно только конкретный именованный объект, относящийся к ПроГео, по точному имени
   (например `docker volume rm progeo_pgdata`), и только после того как: показана точная
   команда → показано, что именно будет удалено (`ls`, `docker volume ls`, `SELECT count(*)`
   и т.п.) → пользователь явно подтвердил именно эту команду словом «да».
3. Общее подтверждение auto-approve НЕ распространяется на п.1–2 — сначала показать, потом
   спросить, дождаться явного «да», даже если операция кажется очевидно безопасной.
4. Если сомнительно, попадает ли команда под эти правила — считать, что попадает, и спросить.

## Проверка результата
- После написания любого эндпоинта показывать как протестировать через curl или написать простой тест
- После изменений запускать линтер/тесты если настроены
- **`tsc --noEmit` и `npm run lint` фронтенда — ТОЛЬКО через `docker compose exec frontend ...`,
  НИКОГДА на хосте.** На хосте `node_modules` неполный/не тот (не резолвятся `maplibre-gl`,
  `@radix-ui/react-slot`, `clsx` и др.) — хостовый `tsc` выдаёт ложные ошибки и может пропускать
  реальные, его вывод недостоверен (найдено 2026-07-16 при разборе ложной ошибки `variant` на
  `Button`, подтверждено `git stash` внутри контейнера).
- **После `makemigrations` ОБЯЗАТЕЛЬНО `migrate` на dev-БД в контейнере, в тот же заход.**
  `makemigrations` без `migrate` — тихая дыра: юнит-тесты пересоздают тестовую БД из всех миграций
  и проходят зелёным, а живая dev-БД остаётся без новой колонки — первый же реальный запрос падает
  500 (найдено 2026-07-16 на `return_note`: `manage.py test` был 83/83, а `POST /requests/` на
  dev-БД упал `column "return_note" does not exist`, потому что забыли `migrate`).

## Выкладка файлов в чат

Когда пользователь просит показать содержимое файла или diff — выкладывать ТЕКСТ
в ответ. Инструмент Read помещает файл в мой контекст, но НЕ показывает его
пользователю. Фразы «выкладываю целиком», «вот полное содержимое», «diff показан
выше» без самого текста — не выкладка.

Для untracked-файлов, где git diff пуст, это особенно важно: единственный способ
показать такой файл — вывести его текст в ответ.

(Найдено 2026-07-20: три раза подряд в одной сессии текст «вот полное содержимое» шёл
без фактического вывода файла — пользователь трижды переспрашивал, прежде чем это
было исправлено.)

## Ключевые инварианты (не нарушать ни на каком этапе)
1. Отзыв возможен только после подтверждённой заказчиком сделки.
2. Статус «принято» ставит только заказчик.
3. Объект (`site`) — первоклассная сущность; заявки/сделки/договоры/файлы привязаны к нему.
4. Монетизация — за feature-toggle, выключена до достижения порога ликвидности.
5. Верификация исполнителя — опциональный процесс: неверифицированный откликается без
   ограничений, статус верификации виден заказчику в каждом отклике.
6. Выбор исполнителя возможен только из рассмотренных заказчиком откликов (минимум один
   рассмотренный обязателен).
7. Заявку можно отозвать только пока на неё нет ни одного отклика; после первого отклика —
   только через администратора.
8. Отзыв и оценка — полностью опциональны, доступны только после статуса «закрыта».
9. До закрытия сделки исполнитель не может узнать, есть ли отклики на заявку (кроме
   собственного) — ни через статус, ни через `updated_at`, ни любым другим полем/каналом.
   Защита от манипуляции ценами: знание о наличии/отсутствии конкурентов искажает
   предложение цены/срока в обе стороны. После перехода заявки в `accepted` торг окончен,
   и факт победы перестаёт быть скрытым: отзыв заказчика публикуется с его именем и
   привязкой к исполнителю (PRODUCT_SPEC 1.10).
10. Подтверждение почты — мягкая блокировка: вход разрешён сразу после регистрации, но создание
    заявки (заказчик) и отклик (исполнитель) заблокированы до подтверждения почты.
11. Все пользовательские строки — через i18n (kk/ru/en).
12. Модули НЕ общаются по HTTP (это монолит); только публичные сервисы в памяти + доменные события.
